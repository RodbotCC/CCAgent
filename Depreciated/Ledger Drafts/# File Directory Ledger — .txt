# File Directory Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **File Directory Ledger** explains the shape of the file tree.

It is the project’s table of contents, directory map, and “where does this belong?” guide.

The Global Ledger tells agents what the project is.
The File Directory Ledger tells agents where they are.

The uploaded ledger plan describes this ledger as the project’s table of contents. It should map the main folders, what they are responsible for, and how they relate to each other. It should prevent agents from finding similar words somewhere else and editing the wrong area. 

That is the core job.

This ledger exists to stop “grep roulette.”

A future agent should not have to blindly search the repo, guess what a folder means, and hope it edits the right thing.

---

# 1. Core Operating Principle

The File Directory Ledger should enforce this rule:

> Before editing a file, understand where it lives and what its directory owns.

The file tree is not just storage.

The file tree is architecture.

Directories should communicate:

* what they own
* what they do not own
* what source of truth they trust
* what local rules apply
* what nearby files are generated
* what nearby files are canonical
* what areas are dangerous
* what agents should read first

The project’s philosophy says the repo is memory. The File Directory Ledger makes the repo navigable memory.

# 3. Role in the Ledger System

The File Directory Ledger sits between global project memory and local Box memory.

```text id="qkt4jq"
Global Ledger
  └── What is the project?

File Directory Ledger
  └── What directories exist?
  └── What does each directory own?
  └── Where should agents go?

Directory Configuration Ledgers / Boxes
  └── What are the local rules inside this directory?

File Contents Ledger
  └── What do important files do?
```

Think of it like this:

```text id="wqwcd2"
Global Ledger = world map
File Directory Ledger = city map
Directory Config / Box = neighborhood sign
File Contents Ledger = building directory
```

The File Directory Ledger should not replace local directory configs. It should point to them.

# 4. What This Ledger Owns

## Owns

* top-level directory map
* important subdirectory map
* directory ownership/responsibility
* canonical vs generated areas
* edit boundaries
* source-of-truth notes by directory
* read-first recommendations
* relationships between directories
* “do not edit here unless…” warnings
* directory status:

  * active
  * generated
  * archive
  * legacy
  * experimental
  * deprecated
  * external
* links to local Box / orientation files
* directory-level Done Gate triggers

## Does not own

* every file summary
* every component dependency
* every local rule
* detailed widget/API maps
* all open problems
* every generated artifact
* full source-of-truth matrix

Those belong in File Contents Ledger, Asset/Widget Map, Source-of-Truth Ledger, local Boxes, Page Ledgers, and Open Problems Ledger.

The File Directory Ledger should be navigational, not encyclopedic.

## 6.2 Directory Philosophy

Purpose: explain how the project thinks about folders.

Example:

```markdown id="2zgj1v"
## 1. Directory Philosophy

The file tree is the project’s working memory.

Directories are not passive containers. Important directories should own a clear area of the system and eventually introduce themselves through a local Box, directory orientation file, or local ledger.

The File Directory Ledger gives project-wide orientation. Local Box files give directory-specific operating rules.

Before editing in an unfamiliar area:
1. Find the directory in this ledger.
2. Read its ownership and source-of-truth notes.
3. Read any local Box / AGENTS / CLAUDE / README file.
4. Edit only after the directory’s role is clear.
```

This sets the behavior.

## 6.3 Read-First Directory Rules

Purpose: give agents immediate safe-navigation rules.

Example:

```markdown id="otldvi"
## 2. Read-First Directory Rules

- Do not edit a directory just because a keyword search found it.
- Check whether the file is canonical, generated, legacy, or archive.
- If a directory has a local Box or orientation file, read it before editing.
- If changing a page, check the page-asset sitemap.
- If changing client-specific truth, work inside `Auto/Client Boxes/<Name>/`.
- If changing runtime rendering, inspect `Auto/orchestrator/`.
- If changing send rules, inspect inbox guardrails before plan text.
- If changing generated state, identify the source generator first.
- If moving or renaming directories, update this ledger.
```

This prevents wrong-area edits.

# 10. Canonical vs Generated Areas

Purpose: prevent agents from editing output as input.

Example:

```markdown id="6o98ht"
## 6. Canonical vs Generated Areas

### Canonical areas

These are usually source-of-truth inputs:

- `LEDGERS/`
- `Auto/Client Boxes/<Name>/`
- `Auto/comeketo-inbox/references/guardrails.md`
- page-asset sitemap
- source scripts/components/routes

### Generated or derived areas

These are usually outputs and should not be hand-edited unless explicitly marked:

- `Auto/orchestrator/state/`
- generated dashboard JSON/CSV
- rendered HTML pages
- build artifacts
- cache-busted bundles
- temporary run outputs

Rule:
If a file looks generated, find the generator before editing.
```

The Source-of-Truth Ledger will later define this more formally.

The File Directory Ledger should give the navigational warning.

# 11. Active / Legacy / Archive Status

Purpose: reduce confusion between current and old systems.

Example:

```markdown id="ycsgpd"
## 7. Active / Legacy / Archive Status

Each significant directory should have a status:

- `active`: current system, safe to modify with proper checks
- `canonical`: source of truth
- `generated`: derived output, do not hand-edit
- `legacy`: old system, inspect before reusing
- `archive`: preserved for history, do not treat as current
- `experimental`: not production truth
- `deprecated`: should not receive new work
- `external`: imported or vendor-owned

Status should be visible in the directory map.
```

Auto/comeketo-inbox      active unpacked skill

# 14. Common Wrong-Turns

Purpose: explicitly name mistakes agents are likely to make.

Example:

```markdown id="bq9z1f"
## 10. Common Wrong-Turns

### Mistake: Editing orchestrator state as truth

Fix:
Find the source Client Box or generator script.

### Mistake: Treating the Boxes page as the client source of truth

Fix:
Boxes page is a UI surface. Client Boxes are canonical.

### Mistake: Editing a seven-day plan without checking guardrails

Fix:
Read Inbox Guardrails and current comms first.

### Mistake: Using enrichment facts as customer-facing facts

Fix:
Check comms/transcripts or allowed-to-know layer.

### Mistake: Updating a page without updating the sitemap

Fix:
Update page-asset source of truth before done.

### Mistake: Fixing one client by changing global automation

Fix:
Prefer local Box fixes unless the pattern is systemic.
```

This section is incredibly useful for future agents.

It should be direct and memorable.

# 15. Directory Change Done Gate

Purpose: define when this ledger must be updated.

Example:

```markdown id="8kdx94"
## 11. Directory Change Done Gate

Update this ledger when:

- a top-level directory is added, removed, renamed, or repurposed
- an important subdirectory is added, removed, renamed, or repurposed
- a directory’s canonical/generated/legacy status changes
- source-of-truth ownership moves between directories
- a local Box or orientation file is added
- a major relationship between directories changes
- agents have repeatedly edited the wrong area
- a new common wrong-turn is discovered
```

Also include:

```markdown id="fn2nl8"
Do not update this ledger for every tiny file addition.
Use the File Contents Ledger for important-file responsibilities.
```

# 24. Quality Bar

The File Directory Ledger succeeds if a future agent can answer:

* What directories exist?
* What does each major directory own?
* What does each major directory not own?
* Where should I go before editing?
* What is canonical?
* What is generated?
* What is legacy/archive?
* What local orientation should I read?
* What common wrong turns should I avoid?

It fails if:

* it becomes an exhaustive file dump
* it is not updated when directories change
* it duplicates page/widget/file ledgers
* it does not distinguish canonical from generated
* it lets agents keep editing the wrong area
* it is too vague to guide actual work
* it invents structure not present in the repo

This ledger should make the repo feel navigable.
