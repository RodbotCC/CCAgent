# Directory Configuration Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Directory Configuration Ledger** gives each meaningful directory its own local operating instructions.

The File Directory Ledger maps the project’s file tree from above.

The Directory Configuration Ledger creates the local “you are here” sign inside important directories.

The uploaded ledger plan describes this as the local orientation layer every meaningful directory should have. It should explain what the directory is for, what it owns, what it does not own, what important files live there, what source of truth applies, and what should be updated when the directory changes. 

This is the first ledger where the project’s **Box** concept really starts to become operational.

A directory should not be a mystery.

A future agent should be able to enter a folder, read one local file, and understand how to behave.

---

# 1. Core Operating Principle

The Directory Configuration Ledger should enforce this rule:

> Every important directory should introduce itself.

A directory configuration should answer:

* Where am I?
* What does this directory own?
* What does this directory not own?
* What is canonical here?
* What is generated here?
* What should never be edited directly?
* What files matter most?
* What local guardrails apply?
* What other directories does this connect to?
* What ledgers must be updated when work happens here?
* What should future agents read before making changes?

This prevents agents from treating every folder like generic storage.

The project wants place-based continuity.
This ledger makes the file tree self-explaining.

# 3. Recommended Local File Names

Each meaningful directory should eventually contain one of these local orientation files.

Preferred:

```text
BOX.md
```

Alternative if the directory is not quite a Box:

```text
DIRECTORY.md
```

Alternative for agent-specific instructions:

```text
AGENTS.md
CLAUDE.md
```

Recommended standard:

```text
<important-directory>/
  BOX.md
```

Use `BOX.md` when the directory has real local memory, rules, source-of-truth notes, or workflow.

Use `DIRECTORY.md` only for simpler orientation.

For this project, most meaningful directories are really Boxes.

# 4. Role in the Ledger System

The Directory Configuration Ledger sits below the File Directory Ledger.

```text
Global Ledger
  └── What is this project?

File Directory Ledger
  └── What directories exist?
  └── What does each directory broadly own?

Directory Configuration Ledger
  └── What should every important directory explain locally?

Local BOX.md
  └── How do I work safely inside this directory?
```

The global Directory Configuration Ledger defines the template and rules.

Each local `BOX.md` applies those rules to a specific place.

# 5. What This Ledger Owns

## Owns

* standard structure for directory configs
* required sections for `BOX.md`
* rules for when a directory needs a config
* local source-of-truth expectations
* local Done Gate expectations
* directory onboarding pattern
* directory status labels
* local memory conventions
* relationship between global ledgers and local Boxes
* how agents should read local orientation before editing

## Does not own

* full file tree map
* every important file summary
* page/widget dependency maps
* detailed client plans
* complete open-problem inventory
* every individual directory’s local content

Those belong in local `BOX.md` files, File Contents Ledger, Page Ledgers, Widget Ledgers, Source-of-Truth Ledger, and Open Problems Ledger.

This ledger defines the **pattern**.
Local Boxes hold the **specifics**.

# 7. When a Directory Needs a Config

Not every folder needs a `BOX.md`.

A directory needs a local config when it meets one or more of these conditions:

```markdown
## 2. When a Directory Needs a Config

Create a local `BOX.md` when a directory:

- owns a major system
- contains canonical source-of-truth files
- contains generated files agents might accidentally edit
- has local rules or guardrails
- has multiple important subfolders
- is frequently edited by agents
- has caused confusion before
- connects to external services
- contains client/customer data
- contains automation or scheduled-task behavior
- contains UI route/page/widget ownership
- requires a local Done Gate
- has local open problems or audit history
```

Examples that clearly need local configs:

```text
/LEDGERS/
/Auto/
/Auto/Client Boxes/
/Auto/orchestrator/
/Auto/comeketo-inbox/
UI/app root
```

Client folders themselves may also need smaller local notes, but the pattern can start with audit markers and grow into `BOX.md` when needed.

# 9. Local `BOX.md` Section Details

## 9.1 What This Directory Is

Example:

```markdown
## 1. What This Directory Is

This directory is the canonical per-client memory substrate for Comeketo leads.

Each child folder is a Client Box: a local memory and operating package for one lead.
```

This should be plain and immediate.

---

## 9.2 What This Directory Owns

Example:

```markdown
## 2. What This Directory Owns

This directory owns:

- per-client comms
- per-client state
- per-client profiles
- seven-day plans
- Andre alert plans
- client audit markers
- allowed-to-know facts when implemented
- local client continuity
```

Ownership should be explicit.

---

## 9.3 What This Directory Does Not Own

This is just as important.

Example:

```markdown
## 3. What This Directory Does Not Own

This directory does not own:

- global inbox guardrails
- UI rendering logic
- generated dashboard state
- global project ledgers
- Close API credentials
- app route ownership
```

This prevents scope bleed.

---

## 9.4 Source-of-Truth Rules

Example:

```markdown
## 4. Source-of-Truth Rules

For client facts, trust in this order:

1. full transcripts, when present
2. `01_comms.md`
3. `00_meta.json`
4. `client_ledger.md`
5. operator-approved notes
6. profile/enrichment only as internal strategy

Seven-day plans are strategy drafts. They do not override comms, state, guardrails, or approvals.
```

Every important directory should explain what to trust.

## 5. Important Files

## 9.7 Generated / Do-Not-Edit Areas

Example:

```markdown
## 7. Generated / Do-Not-Edit Areas

Do not hand-edit generated files unless the local process explicitly allows it.

Generated or derived files:
- `client_ledger.md` may be regenerated by sweep.
- orchestrator state outputs are generated outside this directory.

If a generated file is wrong, fix the source or generator first.
```

This section is essential for agents.

## 9.8 Local Guardrails

Example:

```markdown
## 8. Local Guardrails

- Do not send or approve customer-facing copy from enrichment alone.
- Do not reuse fee-waiver or discount language without explicit approval.
- If a lead replies, pause/rewrite the plan.
- Calendar labels must resolve to explicit dates.
- Client-facing copy must pass Inbox Guardrails.
```

This is where global rules become local.

---

## 9.9 How To Work Here

Example:

```markdown
## 9. How To Work Here

Before editing:

1. Read this `BOX.md`.
2. Read the relevant client folder.
3. Check current comms/state.
4. Check guardrails.
5. Identify whether the target file is source or generated.
6. Make the narrowest safe change.
7. Leave an audit marker if client strategy/state changed.
8. Update relevant ledgers if the directory meaning changed.
```

This is the local agent protocol.

---

## 9.10 What To Update When Work Happens

Example:

```markdown
## 10. What To Update When Work Happens

If you change a client plan:
- update local audit marker
- consider Temporal Continuity if cross-client workflow changed

If you change directory structure:
- update File Directory Ledger
- update this BOX.md

If you change local source-of-truth rules:
- update Source-of-Truth Ledger
- update this BOX.md

If you find an unresolved local issue:
- add to Open Local Problems
- update Open Problems Ledger if project-wide
```

This turns local configs into living memory.

# 10. Directory Status Labels

The Directory Configuration Ledger should standardize labels.

```markdown
## 4. Directory Status Labels

Use one or more:

- `canonical`: source of truth lives here
- `active`: current working system
- `generated`: output from scripts/renderers
- `mixed`: contains both source and generated files
- `legacy`: old but may still contain useful references
- `archive`: historical; do not use as current truth
- `experimental`: not production truth
- `deprecated`: should not receive new work
- `external`: vendor/imported/dependency-owned
- `sensitive`: contains credentials, private data, or risky client info
```

Examples:

```text
Auto/Client Boxes        canonical + active + sensitive
Auto/orchestrator        active + mixed
Auto/orchestrator/state  generated
Auto/comeketo-inbox      canonical + active
LEDGERS                  canonical + active
```

Opus should verify before final labeling.

# 14. Local Done Gate Rules

Each `BOX.md` should include the local Done Gate.

Template:

```markdown
## Local Done Gate

Work in this directory is not done until:

- [ ] correct local source was changed
- [ ] generated files were not edited as source
- [ ] local rules were followed
- [ ] relevant ledgers/maps were updated
- [ ] local open problems were recorded
- [ ] handoff/audit note was left if needed
```

For Client Boxes:

```markdown
Client Box work is not done until an audit marker is added or updated when strategy/state changes.
```

For UI:

```markdown
Page work is not done until the page-asset sitemap is updated.
```

The uploaded sitemap already establishes this rule for page/route/data-binding changes, so local UI directory config should point to it.

# 15. Local Agent Protocol

Each `BOX.md` should contain a simple local work protocol.

Universal pattern:

```text
Read
Locate truth
Check rules
Edit narrowly
Verify
Record
Handoff
```

Markdown version:

```markdown
## Local Agent Protocol

1. Read this `BOX.md`.
2. Read linked local ledgers.
3. Identify canonical source files.
4. Check generated/do-not-edit warnings.
5. Check local guardrails.
6. Make the smallest safe change.
7. Verify or explain why verification was not possible.
8. Update local memory.
9. Update global ledger only if the change affects global understanding.
10. Commit/push or explain local-only status.
```

This should be copied into each important Box, then customized.

# 16. Common Mistakes This Ledger Prevents

This global ledger should name the big mistakes:

```markdown
## 10. Common Mistakes

- Creating a folder with no orientation.
- Letting a directory’s purpose drift without updating its config.
- Treating a local exception as a global rule.
- Editing generated output because the source path was unclear.
- Making agents infer source-of-truth rules from scattered files.
- Overloading Global Ledger with details that belong in a local Box.
- Creating local rules without linking them back to global ledgers.
- Forgetting that directories can contain sensitive client or automation context.
```

# 19. Relationship to Boxes

This ledger is where the Box concept becomes concrete.

A Box is:

```text
A directory with memory, rules, source-of-truth notes, local ledgers, handoff context, and a Done Gate.
```

A Box is not:

```text
just a folder
just a README
just a doc dump
```

The Directory Configuration Ledger should say:

```markdown
A Box should make a directory agent-operable.
```

That means:

* the agent knows where it is
* the agent knows what the directory owns
* the agent knows what not to touch
* the agent knows what to update
* the agent knows how to leave continuity behind

This is the local intelligence layer of the project.

# 22. Quality Bar

The Directory Configuration Ledger succeeds if:

* important directories introduce themselves
* agents know what a directory owns before editing
* local source-of-truth rules are visible
* generated/do-not-edit areas are clearly marked
* local Done Gates exist where needed
* global ledgers are not overloaded with local detail
* future agents can enter a folder and behave safely
* Box architecture becomes real through files, not just philosophy

It fails if:

* `BOX.md` files become vague READMEs
* local configs are not updated when directory responsibilities change
* every folder gets a config whether it needs one or not
* local configs duplicate global ledgers
* agents still have to guess what a directory does
* generated files remain indistinguishable from source truth

This ledger should make directories feel alive, oriented, and safe.

---

# 23. Final Operating Rule

The Directory Configuration Ledger should end with this:

```markdown
A meaningful directory should not be silent.

If a directory matters, it should say what it is, what it owns, what it trusts, what it forbids, and how to leave it better than you found it.
```
