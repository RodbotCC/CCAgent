# Page Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Page Ledger** tracks the state, responsibilities, data flow, assets, source-of-truth assumptions, Done Gate, and open problems for a specific page or screen.

The Asset / Widget Map Ledger gives the global UI map.

The Page Ledger gives one page its own local memory.

The uploaded ledger plan describes the Page Ledger as the place to answer what a page is for, what widgets live there, what data it uses, what APIs it calls, what source of truth it trusts, what states it supports, what done means for the page, what open problems exist, and what changes recently affected it. 

This is crucial for this project because the UI is not just visual.

Pages like `boxes`, `automation`, `delegations`, `intake`, and `settings` can expose state, write data, trigger workflows, or imply truth. A page that looks simple may be sitting on top of Client Boxes, generated orchestrator state, approval flows, credentials, or external APIs.

The Page Ledger makes each page legible.

---

# 1. Core Operating Principle

The Page Ledger should enforce this rule:

> A page is not done until its purpose, data, source truth, states, side effects, and Done Gate are clear.

A page should answer:

* What is this page for?
* Who uses it?
* What route owns it?
* What component renders it?
* What widgets live here?
* What data does it read?
* What does it write?
* What APIs does it call?
* What source of truth does it trust?
* What states can it be in?
* What side effects can it trigger?
* What breaks if this page changes?
* What must be updated when this page changes?

The project already has a page-asset sitemap that defines route/page ownership and says page, route behavior, or page data-binding changes are incomplete until the sitemap is updated. The Page Ledger should build on that, not replace it.

# 3. Role in the Ledger System

The Page Ledger sits under the Asset / Widget Map Ledger.

```text
Asset / Widget Map Ledger
  └── global page/widget/API/data map

Page Ledger
  └── page-specific purpose, states, data, APIs, side effects, Done Gate, open issues

Widget Ledger
  └── detailed behavior for complex widgets inside a page

Source-of-Truth Ledger
  └── canonical source rules for data shown on the page

Definition of Done Ledger
  └── page-change completion requirements
```

The relationship is:

```text
Asset / Widget Map = global UI atlas
Page Ledger = page dossier
Widget Ledger = component deep dive
```

# 4. What This Ledger Owns

## Owns

* page purpose
* route/component ownership
* page-level widgets
* page-level data dependencies
* API dependencies
* read/write behavior
* side effects
* supported UI states
* empty/error/loading states
* source-of-truth assumptions
* page-specific Done Gate
* page-specific open problems
* recent page changes
* links to related widgets
* page verification checklist
* page-level audit history

## Does not own

* every tiny component detail
* all shared widget internals
* full backend API implementation
* global source-of-truth policy
* all UI styles
* full route map
* every file description

Those belong in Widget Ledgers, File Contents Ledger, Source-of-Truth Ledger, Asset / Widget Map, and code.

The Page Ledger should keep page-level truth clear without becoming a code duplicate.

# 7. Page Status Labels

```markdown
## 3. Page Status Labels

Use one or more:

- `active`: current live page
- `read-only`: page displays data but should not write
- `write-capable`: page can persist data or trigger actions
- `high-risk`: page can affect automation, credentials, customer-facing actions, or source truth
- `generated-view`: page displays generated/derived state
- `source-editing`: page can directly edit canonical data
- `experimental`: page is not stable
- `legacy`: page exists but should not guide new work
- `needs-audit`: page needs review
```

Examples:

```text
boxes        active + generated-view + high-risk
automation   active + write-capable + high-risk
delegations  active + write-capable + high-risk
settings     active + write-capable + sensitive
intake       active + write-capable + medium/high risk
analytics    active + generated-view
```

Opus should verify exact statuses from current code and sitemap.

# 10. Example Page Record — `boxes`

This should be seeded first because it connects directly to Client Boxes.

````markdown
# Page Ledger — boxes

Last updated: 2026-04-28  
Route: `boxes`  
Primary component: `BoxesScreen`  
Owner file: `screens.jsx`  
Page status: active / generated-view / high-risk  
Risk level: high  
Related sitemap section: Comeketo Page-Asset Source Of Truth — boxes  
Related widgets: Boxes dossier renderer, template preview, delegation handoff action  
Related APIs:
- `/api/boxes/list`
- `/api/boxes/<id>`
- `/api/boxes/<id>/html`
- `/api/boxes/<id>/template/<slug>`

## 1. Page Purpose

The `boxes` page is a dossier/read surface for Box data.

It displays selected Client Boxes, Staff Boxes, and generated orchestrator state in a single UI surface.

It helps operators and agents inspect client/staff context, completeness, comms, profile, plan, alerts, and possible handoff actions.

## 2. User / Operator Use Case

The operator uses this page to understand what a Box contains and whether it is ready, stale, incomplete, or needs action.

For Client Boxes, this page should help answer:

- What is in the client box?
- Is it complete?
- What state does it show?
- What plan/alerts exist?
- What source files back this display?
- Is this view fresh or generated?
- Does this need a handoff/delegation?

## 3. Route + Render Ownership

Route:
`boxes`

Primary owner:
`screens.jsx` / `BoxesScreen`

Backend/API owner:
`server.py` endpoints under `/api/boxes/*`

Page ownership source:
page-asset sitemap.

## 4. Data Sources

Reads:

- `Auto/Client Boxes`
- `Auto/Staff Boxes`
- `Auto/orchestrator/state`
- loader data from `mission_control_loader.js` if applicable

## 5. APIs

- `/api/boxes/list`
- `/api/boxes/<id>`
- `/api/boxes/<id>/html`
- `/api/boxes/<id>/template/<slug>`

## 6. Writes / Side Effects

Expected default:
read/display only.

Possible side effects:
optional delegation draft handoff actions.

Important:
The page should not directly rewrite canonical Client Box truth unless a future explicit edit workflow is built and documented.

## 7. Source-of-Truth Rules

Canonical client truth lives in:

```text
Auto/Client Boxes/<Client Name>/
````

Canonical staff truth lives in:

```text
Auto/Staff Boxes/
```

Generated/rendered state may live in:

```text
Auto/orchestrator/state
```

The Boxes page is not the source of truth.

If displayed client data is wrong, check the Client Box and generator before changing the page.

## 8. Widgets On This Page

Initial widgets/panels may include:

* box list
* selected box dossier
* source file sections
* HTML preview
* template preview
* completeness/status indicators
* delegation handoff action

Each high-risk widget should later get a Widget Ledger entry if it writes data or affects approvals.

## 9. Supported States

The page should support:

* loading box list
* no boxes found
* selected box loading
* selected box loaded
* box missing required files
* source file missing
* generated HTML unavailable
* API error
* stale generated state
* read-only display
* delegation handoff available
* delegation handoff error/success, if implemented

## 10. Empty / Error / Loading States

The page should not silently show blank panels.

Every empty state should say what is missing:

* no boxes found
* no selected box
* source file unavailable
* generated preview unavailable
* API error
* stale state warning if known

## 11. Page-Specific Guardrails

* Do not imply generated state is canonical.
* Do not allow display fixes by editing generated output manually.
* Do not show enrichment as customer-facing truth without classification.
* Do not trigger delegation handoff without explicit operator action.
* If future editing is added, it must obey Client Box source-of-truth rules.

## 12. Page Done Gate

A change to this page is not done until:

* page-asset sitemap updated
* Asset / Widget Map updated if dependencies changed
* Source-of-Truth Ledger updated if page authority changed
* Widget Ledgers updated for new high-risk widgets
* verification performed or honestly marked unavailable

## 13. Verification Checklist

* `/api/boxes/list` returns expected list.
* selecting a box loads expected source files.
* Client Box displayed data matches actual files.
* generated preview is labeled as generated.
* missing file state is readable.
* no write occurs during read-only navigation.
* delegation handoff, if used, writes only expected draft/handoff target.

## 14. Open Page Problems

Seed possible issues:

* Need clearer source/freshness labeling.
* Need explicit canonical/generated distinction.
* Need local handling for allowed-to-know layer once implemented.
* Need verify whether delegation handoff side effects are fully mapped.

## 15. Recent Page Changes

Start empty or seed from sitemap history.

## 16. Related Ledgers / Files

* Page-Asset Sitemap
* Asset / Widget Map Ledger
* Source-of-Truth Ledger
* Client Boxes Directory Config
* File Contents Ledger
* Open Problems Ledger

````

The sitemap already says the `boxes` page reads `Auto/Client Boxes`, `Auto/Staff Boxes`, and `Auto/orchestrator/state`, and that page/route/data-binding work must update the sitemap before completion. :contentReference[oaicite:3]{index=3}
````

# 13. Page Done Gate

Every page ledger should include a page-specific Done Gate, but the global Page Ledger should define the default.

```markdown
## 6. Page Done Gate

A page change is done only when:

- [ ] route still maps correctly
- [ ] primary component ownership is clear
- [ ] visible widgets are accounted for
- [ ] data sources are documented
- [ ] APIs are documented
- [ ] writes / side effects are documented
- [ ] source-of-truth assumptions are clear
- [ ] empty/loading/error states are not broken
- [ ] page-asset sitemap is updated
- [ ] Asset / Widget Map is updated if dependencies changed
- [ ] Page Ledger is updated if page purpose/data/side effects changed
- [ ] verification was performed or limitations are recorded
```

For this repo, the existing sitemap already makes page/route/data-binding updates mandatory before work is complete.

# 14. Page State Standards

Each page should define supported states.

Standard page states:

```markdown
## 7. Page State Standards

Each page should specify whether it supports:

- loading
- loaded
- empty
- missing source
- stale data
- permission/access error
- API error
- partial data
- generated preview unavailable
- write pending
- write success
- write failure
- read-only mode
- dangerous action confirmation
```

High-risk pages must not fail silently.

For example, `boxes` should distinguish:

```text
Client Box missing
vs
Client Box present but incomplete
vs
orchestrator generated state stale
vs
API error
```

# 15. Page Verification Standards

Default verification checklist:

```markdown
## 8. Page Verification Standards

When changing a page, verify:

- route loads
- primary page state renders
- expected data appears
- empty state works
- API error does not crash page
- writes go to expected target
- no unexpected side effects occur
- page still matches sitemap
- source-of-truth assumptions remain true
```

If verification cannot be done:

```markdown
Record what was not verified and why.
```

No fake green checks.

# 18. Relationship to Page-Asset Sitemap

This is the most important relationship.

The Page Ledger should not replace the existing sitemap.

The sitemap is the current operational page/source-of-truth file. It maps route index, asset ownership, APIs, side effects, change checklists, history, and last verified fields. It also states that page, route behavior, or data-binding work is not complete until the relevant sitemap areas are updated. 

The Page Ledger should say:

```markdown
The sitemap is the operational page ownership map.
The Page Ledger gives each page a local dossier and deeper continuity.
When they conflict, audit and update both; do not let them drift.
```

---

# 19. Relationship to Asset / Widget Map

The Asset / Widget Map gives the global view.

The Page Ledger gives the page-specific view.

Example:

```text
Asset / Widget Map:
BoxesScreen reads Client Boxes and orchestrator state.

Page Ledger:
Boxes page must show missing/stale/generated states and must not imply generated state is canonical.
```

The Asset / Widget Map answers “what connects?”
The Page Ledger answers “how should this page behave?”

---

# 20. Relationship to Widget Ledger

A Page Ledger lists widgets on the page.

A Widget Ledger deep-dives into a complex widget.

Example:

```text
Page Ledger — automation:
Lists workflow graph, trigger panel, hooks panel, state panel.

Widget Ledger — workflow graph:
Details graph data model, save behavior, node/edge schema, side effects, validation, common bugs.
```

Create Widget Ledgers only where complexity justifies it.

---

# 21. Relationship to Source-of-Truth Ledger

Each Page Ledger should include source-of-truth rules.

Example for `boxes`:

```text
Displayed client facts:
source = Auto/Client Boxes

Generated HTML:
source = renderer output, not canonical truth

Client plan:
source = Client Box plan, but plan is subordinate to comms/state/guardrails
```

The Source-of-Truth Ledger owns the global authority rules.
The Page Ledger applies them locally.

---

# 22. Relationship to Open Problems

Each Page Ledger can hold local page problems.

If a problem affects many pages or a global pattern, promote it to Open Problems Ledger.

Example:

```text
Local page problem:
Boxes page needs better stale-state label.

Global open problem:
UI pages do not consistently distinguish canonical vs generated data.
```

---

# 23. Relationship to Audit Ledger

Page audits should create/update Page Ledger entries.

Example:

```text
Audit:
Boxes page source/display audit.

Updates:
- Audit Ledger entry
- Page Ledger boxes.md
- Asset / Widget Map if dependencies changed
- Open Problems if unresolved page risks remain
```

---

# 24. Page Change Done Gate

Update the Page Ledger when:

```markdown
- page purpose changes
- route changes
- primary component changes
- data source changes
- API dependency changes
- side effects change
- supported states change
- source-of-truth assumptions change
- high-risk widgets are added/removed
- page is audited
- page-specific open problems are found or resolved
```

Do not update for tiny visual polish unless it affects page ownership, data, source truth, states, or Done Gate.

* What source of truth does it trust?
* What states should it support?
* What does done mean for this page?
* What page-specific problems are open?
* What changed recently?

It fails if:

* it duplicates the sitemap line-by-line
* it ignores side effects
* it does not identify source truth
* it tracks tiny visual details instead of page responsibilities
* high-risk pages lack ledgers
* page changes keep happening without updating it
* it does not connect to Widget Ledgers

This ledger should make pages safe to modify.

---

# 27. Final Operating Rule

The Page Ledger should end with this:

```markdown
A page is not just a screen.

A page is a contract between route, component, data, state, source truth, side effects, and user expectation.

Change the page, update the contract.
```
