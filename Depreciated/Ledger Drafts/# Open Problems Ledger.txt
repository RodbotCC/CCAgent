# Open Problems Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Open Problems Ledger** tracks what is broken, incomplete, risky, unresolved, unclear, blocked, partially fixed, or repeatedly rediscovered.

It is the project’s durable “not solved yet” memory.

The uploaded ledger plan says problems should not disappear just because a session ends. This ledger should answer what is broken, incomplete, risky, blocked, partially fixed, unresolved, or worth revisiting later. It should be used aggressively, because if a known problem is not recorded, the system will waste time finding it again. 

That is the core job:

> Known problems should not have to be rediscovered.

This ledger is not a shame file.

It is a protection system.

A project this complex will always have open problems. The danger is not having problems. The danger is losing track of them.

---

# 1. Core Operating Principle

The Open Problems Ledger should enforce this rule:

> If a problem is known but not fixed, record it.

A problem can be:

* a bug
* a stale file
* a risky plan
* an unclear source of truth
* a missing decision
* a fragile workflow
* an unsafe automation path
* an unresolved design question
* a partial implementation
* an unverified dependency
* a confusing directory
* a local/GitHub drift issue
* a repeated mistake
* a “we need to revisit this” note

If it matters and remains unresolved, it belongs here or in a local Box-level problem section.

# 3. Role in the Ledger System

The Open Problems Ledger sits beside Decisions, Communications, and Audit.

```text id="0u9du3"
Communications Ledger
  └── What should future agents hear?

Open Problems Ledger
  └── What remains unresolved?

Decisions Ledger
  └── What has been decided?

Audit Ledger
  └── What was reviewed and what did we find?

Temporal Continuity Ledger
  └── Which problems matter right now?
```

Important distinction:

```text id="4h0swz"
Communication:
"Be careful — plans may predate guardrails."

Open Problem:
"Audit all active Client Box plans for guardrail drift."

Decision:
"Risky sales moves require isolated approval."

Audit:
"Brenda & Steve plan had stale dates and ungated fee-waiver language."
```

The Open Problems Ledger tracks the unresolved work.

# 4. What This Ledger Owns

## Owns

* known bugs
* unresolved risks
* incomplete implementations
* blocked work
* unclear questions
* stale files needing cleanup
* partial fixes
* repeated failure patterns
* missing ledgers/maps
* missing source-of-truth definitions
* unverified external dependencies
* local problems with global impact
* problems discovered in audits
* review-needed items

## Does not own

* settled decisions
* full audit reports
* every task
* full project roadmap
* all current state
* every local client detail
* every chat note
* implementation recipes

Those belong in Decisions, Audit, Phase, Temporal Continuity, Client Boxes, Communications, and Reconstruction ledgers.

This ledger should be a live board of unresolved issues, not a generic task manager.

# 6. Problem Entry Format

Use stable IDs.

Recommended ID format:

```text id="j4o64o"
PROB-YYYY-MM-DD-###
```

Example:

```text id="qevtnj"
PROB-2026-04-28-001
```

Problem template:

```markdown id="dj2p5e"
## PROB-YYYY-MM-DD-### — <Short Problem Title>

Status:
Severity:
Urgency:
Discovered:
Discovered by:
Affected systems:
Related files:
Related ledgers:
Owner:
Blocked by:

### Problem

What is wrong, incomplete, risky, or unclear.

### Evidence

How we know this is a problem.

### Impact

What breaks or becomes risky if this remains open.

### Current Workaround

How to operate safely until fixed.

### Recommended Next Action

What should happen next.

### Close Criteria

What must be true before this problem can be closed.

### History

- YYYY-MM-DD — note
```

Every problem should have close criteria.

Without close criteria, problems become vague anxiety.

# 7. Problem Status Labels

Standardize statuses.

```markdown id="aqs6n1"
## 3. Problem Status Labels

Use one:

- `open`: known and not fixed
- `triaged`: understood enough to prioritize
- `in-progress`: currently being worked
- `blocked`: cannot proceed until something happens
- `partial`: partially fixed but not closed
- `needs-decision`: requires a decision before work
- `needs-audit`: requires review/investigation
- `needs-verification`: likely fixed but not verified
- `closed`: fixed or no longer relevant
- `wont-fix`: acknowledged and intentionally not fixed
```

Add optional tags:

```text id="xyw129"
tags:
- safety
- client-box
- automation
- UI
- source-of-truth
- git
- connection
- ledger
- generated-state
- enrichment
- approval
```

Tags make the ledger easier to filter.

# 8. Severity / Urgency Scale

Separate severity from urgency.

```markdown id="oa8g0r"
## 4. Severity / Urgency Scale

### Severity

- `critical`: can cause customer-facing harm, data loss, unsafe sends, credential exposure, or broken core workflow
- `high`: can cause major confusion, stale state, wrong source-of-truth edits, or broken automation
- `medium`: important but bounded; workarounds exist
- `low`: cleanup, clarity, polish, or future improvement

### Urgency

- `now`: should be addressed before further related work
- `soon`: should be addressed in current phase
- `later`: important but not blocking current work
- `watch`: monitor; no immediate action
```

Example:

```text id="sgzuhb"
Problem: unapproved fee-waiver automation path
Severity: critical
Urgency: now
```

This prevents every problem from becoming equally loud.

## 9.1 Allowed-to-Know Constraint Not Implemented

```markdown id="ui4fgx"
## PROB-2026-04-28-001 — Allowed-To-Know Constraint Not Implemented

Status: open  
Severity: critical  
Urgency: now  
Discovered: 2026-04-28  
Discovered by: user + audit conversation  
Affected systems: Client Boxes, inbox automation, scheduled fires, profile/enrichment, customer-facing copy  
Related files: `Auto/Client Boxes/<Name>/01_comms.md`, `04_profile.md`, `05_seven_day_plan.md`, future allowed-to-use files  
Related ledgers: Source-of-Truth, Client Boxes, Audit, Decisions  
Owner: TBD  
Blocked by: schema/design decision  

### Problem

The system does not yet have a formal layer that separates:

- facts the client actually shared
- facts from transcripts/comms
- enrichment-only internal strategy
- protected/off-limits facts
- unverified facts
- approval-required facts/actions

### Evidence

Brenda & Steve highlighted the risk: enrichment-derived context was useful internally but could leak into customer-facing copy. The user explicitly said the project needs a “what we’re allowed to know” type of constraint.

### Impact

Without this layer, automation may surface creepy or unsupported personalization, misuse enrichment, or treat internal strategy as customer-facing truth.

### Current Workaround

During Client Box audits, manually flatten outbound copy and only use facts traceable to comms/transcripts or explicit approval.

### Recommended Next Action

Design a Client Box-level `allowed_to_use` schema and workflow.

Possible files:
- `allowed_to_use.md`
- `allowed_to_use.json`
- `protected_facts.md`
- `approval_required.md`

### Close Criteria

- Schema exists.
- At least one Client Box uses it.
- Scheduled-fire/read protocol knows to check it.
- Guardrails reference it.
```

This is likely one of the most important open problems.

## 9.3 Approval UI Does Not Yet Isolate Risky Moves

```markdown id="t6rs1u"
## PROB-2026-04-28-003 — Approval Flow Does Not Yet Isolate Risky Moves

Status: open  
Severity: critical  
Urgency: soon  
Affected systems: approval surfaces, inbox automation, scheduled fires, Delegations/Boxes if used for review  
Related decisions: DEC-2026-04-28-005 Risky Sales Moves Require Isolated Approval  

### Problem

Risky moves can be approved as part of a batch without enough explicit friction. The system needs a clear approval-card pattern for financial, pricing, scope, enrichment, or high-risk personalization moves.

### Evidence

The Brenda & Steve fee-waiver move was technically approved by the user, but the user was approving many clients quickly while being rushed.

### Impact

The system can make strategically plausible but authority-sensitive moves without the human fully noticing the risk.

### Current Workaround

During plan cleanup, flatten risky future moves or mark them approval-required.

### Recommended Next Action

Design explicit risk approval cards with required confirmations such as:
- `APPROVE FEE WAIVER`
- `APPROVE ENRICHMENT USE`
- `APPROVE PRICE COMMITMENT`
- `APPROVE SCOPE PROMISE`

### Close Criteria

- Approval-card standard exists.
- Risk categories are defined.
- Approval is logged.
- Scheduled fires check approval before risky sends.
```

## 9.4 GitHub Direct Writes Can Conflict With Local Unpushed Work

```markdown id="b6dqgk"
## PROB-2026-04-28-004 — GitHub Direct Writes Can Conflict With Local Unpushed Work

Status: open  
Severity: medium  
Urgency: soon  
Affected systems: GitHub workflow, agent writes, user local repo  
Related ledgers: Connections, Communications, Decisions  

### Problem

Agents may write directly to GitHub while the user has local unpushed work.

### Evidence

After Brenda & Steve cleanup commits, the user realized they had local work not yet pushed and asked how to avoid messing up agent commits.

### Impact

The user may face merge conflicts, rebase confusion, or accidental overwrites if branch workflow is unclear.

### Current Workaround

Use safe Git process:
- create branch
- commit local work
- fetch
- rebase or merge remote
- avoid force-pushing main

### Recommended Next Action

Define standard branch workflow for agent writes:
- direct-to-main only when user confirms local tree is clean
- otherwise agent writes to branch/PR
- report commit SHAs
- avoid force pushes

### Close Criteria

- Git workflow policy recorded.
- Agents know when to write direct-to-main vs branch.
- Connections/Communications ledgers updated.
```

# 15. Problem Lifecycle

Define the lifecycle.

```markdown id="d6sw8s"
## 11. Problem Lifecycle

Problem discovered
→ record it
→ triage severity/urgency
→ assign system/owner if possible
→ define workaround
→ define close criteria
→ work/fix/audit
→ verify
→ close or mark partial
→ archive if no longer useful
```

Mermaid:

```mermaid id="dc4a68"
flowchart TD
  Found[Problem Found] --> Record[Record Problem]
  Record --> Triage[Triage Severity / Urgency]
  Triage --> Workaround[Define Workaround]
  Workaround --> Next[Recommended Next Action]
  Next --> Fixed{Fixed?}
  Fixed -->|No| Open[Keep Open / Blocked / Partial]
  Fixed -->|Yes| Verify[Verify]
  Verify --> Close[Close With Evidence]
```

# 21. Relationship to Definition of Done

A meaningful task is not done if it discovers an open problem and fails to record it.

Definition of Done should reference:

```markdown id="0x51gg"
If you found a problem and did not fix it, record it in Open Problems or a local Box.
```

This turns problem discovery into durable progress.

# 22. Local vs Global Problems

Not every problem belongs globally.

Use this rule:

```text id="2duj1v"
If the problem affects one folder/client only, record it locally.
If it affects a pattern, workflow, source-of-truth rule, safety posture, or multiple systems, record it globally.
```

Examples:

```text id="384vqh"
Local:
Brenda & Steve plan needed cleanup → local audit marker.

Global:
Many Client Box plans may predate guardrails → Open Problems Ledger.
```

This keeps the global ledger from becoming noisy.

# 24. Quality Bar

The Open Problems Ledger succeeds if future agents can answer:

* What is currently broken?
* What is incomplete?
* What is risky?
* What needs a decision?
* What is blocked?
* What was partially fixed?
* What keeps getting rediscovered?
* What is the workaround?
* What would close the problem?
* Which system owns the problem?

It fails if:

* problems stay in chat only
* it becomes a generic task list
* problems lack close criteria
* severity/urgency are missing
* closed problems are not verified
* local-only problems clutter the global ledger
* future agents rediscover the same issues repeatedly
* scary problems are softened into vague notes

This ledger should make unresolved work visible, not overwhelming.

# 25. Final Operating Rule

The Open Problems Ledger should end with this:

```markdown id="922ccw"
A known problem that is not recorded becomes future confusion.

If it is broken, name it.
If it is risky, classify it.
If it is blocked, say why.
If it is fixed, prove it.
```
