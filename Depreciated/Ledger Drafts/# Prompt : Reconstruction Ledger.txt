# Prompt / Reconstruction Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Prompt / Reconstruction Ledger** records how important parts of the system were created, regenerated, rebuilt, or conceptually shaped.

It is not just chat history.

It is rebuild memory.

The uploaded ledger plan describes this ledger as the place that captures the intent, instructions, constraints, reasoning, and rough recipe behind major parts of the system so they can be rebuilt, regenerated, audited, or upgraded later. It should answer what we were trying to build, what instructions shaped it, what constraints mattered, what should be preserved if rebuilt, what should not be repeated, what files/tools/APIs/models/ledgers were involved, and how a future agent could recreate it. 

That is the heart of it:

> Code is the artifact.
> The Reconstruction Ledger is the recipe.

This ledger makes the project reconstructable.

---

# 1. Core Operating Principle

The Prompt / Reconstruction Ledger should enforce this rule:

> If a future agent may need to rebuild it, preserve the recipe.

The project should not depend on invisible generation history.

If a page, workflow, Client Box plan, data pipeline, automation, ledger system, or UI pattern was created through a prompt-driven process, the project should preserve enough context to recreate it later with a better model, a safer process, or a new implementation.

The goal is not to save every prompt.

The goal is to preserve the instructions and constraints that shaped important artifacts.

# 3. Role in the Ledger System

The Reconstruction Ledger is closely tied to Decisions, Communications, Audit, and File Contents.

```text id="ie52ag"
Decisions Ledger
  └── Why did we choose this direction?

Reconstruction Ledger
  └── How would we rebuild this artifact/system?

File Contents Ledger
  └── What files does the artifact use?

Audit Ledger
  └── What did review find after it was built?

Prompt / Reconstruction Ledger
  └── What prompt, constraints, recipe, and model context produced it?
```

This ledger turns ephemeral generation into durable capability.

---

# 4. What This Ledger Owns

## Owns

* build recipes
* generation prompts or summarized prompt instructions
* reconstruction instructions
* artifact intent
* constraints used during generation
* source files involved
* output files created
* model/tool context where useful
* known mistakes to avoid on rebuild
* regeneration triggers
* preservation requirements
* upgrade paths
* links to commits and audits

## Does not own

* every chat message
* every tiny code change
* every local task
* full implementation docs for every file
* every design decision
* all open problems
* all source-of-truth rules

Those belong in Communications, Decisions, File Contents, Open Problems, Source-of-Truth, and local Box files.

This ledger should preserve reconstruction value, not transcript noise.

# 7. What Deserves Reconstruction Memory

Not everything needs an entry.

Create a reconstruction entry when the work is:

```markdown id="pk8kzt"
## 3. What Deserves Reconstruction Memory

Record reconstruction memory for:

- generated pages or UI systems
- generated Client Box plans
- guardrail systems
- ledger system architecture
- automation workflows
- prompt-heavy artifacts
- complex refactors
- data pipelines
- generated reports
- model-authored docs that become source-of-truth
- templates used repeatedly
- systems likely to be regenerated with better models
- anything expensive or dangerous to recreate from scratch
```

Do not record:

```markdown id="8nkwuw"
- tiny edits
- ordinary typo fixes
- one-off small changes
- raw chat logs with no rebuild value
- secrets
```

## 8.2 Client Box Cleanup Reconstruction Recipe

```markdown id="0j6p5l"
## RECON-2026-04-28-002 — Client Box Audit / Cleanup Pass

Status: active  
Created: 2026-04-28  
Artifact type: repeatable audit workflow  
Affected systems: `Auto/Client Boxes`, inbox guardrails, scheduled fires  
Output pattern: cleaned plan/alerts + audit marker  

### Intent

Create a repeatable workflow for auditing and cleaning Client Boxes one at a time so automation can create safe movement without blindly trusting stale or risky plan text.

### Inputs

For each Client Box:

- `00_meta.json`
- `01_comms.md`
- full transcripts when available
- `04_profile.md`
- `05_seven_day_plan.md`
- `09_andre_alerts.md`
- `client_ledger.md`
- Inbox Guardrails
- page/source maps if UI is affected

### Constraints

- Work one box at a time.
- Report findings before moving on unless user authorizes cleanup.
- Do not treat plan as send license.
- Check calendar reality.
- Check reply gate.
- Check frequency cap.
- Check commitment language.
- Check enrichment boundary.
- Check customer-facing facts against comms/transcripts.
- Leave audit marker after cleanup.
- Do not globally rewrite automation for one client’s local issue unless pattern is systemic.

### Build / Generation Recipe

For each box:

1. Confirm required files exist.
2. Read meta and current state.
3. Read comms and transcript truth.
4. Read profile but treat as internal strategy.
5. Read seven-day plan and alerts.
6. Check client ledger freshness.
7. Compare plan timing to actual dates.
8. Flag risky moves:
   - fee waiver
   - discount
   - no charge
   - guest-count promise
   - pricing guarantee
   - scope expansion
   - enrichment-based personalization
9. Flatten future copy or gate risky moves.
10. Align alerts with plan.
11. Leave audit marker with timestamp, files reviewed, files changed, issues found, remaining risks.
12. Report summary and commit SHAs if GitHub writes occur.

### Preservation Requirements

- Human-approved historical sends remain history.
- Future risky language must require approval.
- Client-specific lessons remain in the client Box.
- Systemic lessons move to Open Problems / Decisions / Guardrails.

### Known Mistakes To Avoid

- Rewriting history.
- Deleting useful strategy instead of making it internal-only.
- Treating enrichment as customer-facing truth.
- Fixing generated state instead of source files.
- Forgetting audit marker.
```

This recipe is immediately useful for continuing Client Box work.

## 8.3 Allowed-To-Know Layer Reconstruction Recipe

````markdown id="924pbq"
## RECON-2026-04-28-003 — Allowed-To-Know / Allowed-To-Say Layer

Status: planned  
Created: 2026-04-28  
Artifact type: safety/source-of-truth schema  
Affected systems: Client Boxes, inbox automation, scheduled fires, enrichment, profiles  

### Intent

Create a formal layer in each Client Box that separates what the system knows internally from what it may safely use in customer-facing copy.

### Inputs

- full comms
- phone transcripts
- `01_comms.md`
- `04_profile.md`
- enrichment outputs
- Inbox Guardrails
- human approval notes
- Client Box audits

### Proposed Categories

- `client_shared`: facts directly shared by the client in SMS/email/call/intake
- `operator_shared`: facts Andre/Rhonna logged from real conversation
- `safe_operational`: event details safe to use
- `internal_strategy`: enrichment or inference for internal tone only
- `protected`: never use customer-facing
- `unverified`: do not use until confirmed
- `approval_required`: may be used only with explicit approval
- `risky_action`: actions like fee waivers/discounts/scope promises

### Possible Files

```text
Auto/Client Boxes/<Name>/allowed_to_use.md
Auto/Client Boxes/<Name>/allowed_to_use.json
Auto/Client Boxes/<Name>/protected_facts.md
Auto/Client Boxes/<Name>/approval_required.md
````

### Build / Generation Recipe

1. Parse `01_comms.md` and transcripts.
2. Extract client-shared facts with citations to comms/transcript line/activity.
3. Extract operational event facts.
4. Identify enrichment/profile claims.
5. Classify each fact.
6. Generate `allowed_to_use.md/json`.
7. Require scheduled-fire composition to use only allowed facts unless approval exists.
8. Store approval references when risky facts/actions are cleared.

### Preservation Requirements

* Do not erase enrichment; classify it.
* Do not let internal strategy leak into copy.
* Do not use facts without provenance.
* Co-decider facts count if shared on the thread/call.

````

This will become a major safety unlock.
````

# 9. Prompt Preservation Rules

This ledger should define when raw prompts should be saved.

```markdown id="9zo8r2"
## 6. Prompt Preservation Rules

Preserve raw prompts only when they have rebuild value.

Prefer clean reconstruction recipes over massive raw chat dumps.

Save:

- original high-level instructions
- critical constraints
- examples that shaped the output
- prompts that can regenerate an artifact
- model-specific caveats if relevant

Do not save:

- secrets
- private tokens
- irrelevant chat
- emotional back-and-forth with no rebuild value
- huge transcripts unless they are source data and belong elsewhere
```

A good reconstruction entry should make raw prompt history optional.

```markdown id="rlahtl"
## 7. Rebuild Quality Rules

- understand intent
- gather inputs
- preserve constraints
- avoid known mistakes
- regenerate the artifact
- verify the result
- know what ledgers/maps need updating
- know when not to rebuild
```

# 11. Known Regeneration Risks

Seed these risks:

```markdown id="4q84xf"
## 8. Known Regeneration Risks

### Risk: old prompts may predate current guardrails

Any prompt that generated seven-day plans before Inbox Guardrails v2.0 may produce unsafe commitment language or enrichment leakage.

### Risk: model may over-personalize from enrichment

Regeneration must respect allowed-to-know constraints.

### Risk: generated UI may drift from sitemap

Any regenerated page must update the page-asset sitemap.

### Risk: ledger regeneration may become generic

Ledgers must preserve the project’s actual philosophy, not generic documentation templates.

### Risk: local source truth may change

Before regenerating artifacts, check current Source-of-Truth Ledger and local Box configs.
```

# 20. Relationship to Prompt Privacy / Safety

Do not store raw prompts that include:

* secrets
* private credentials
* irrelevant sensitive information
* unneeded client data beyond what belongs in the client Box
* model chain-of-thought
* internal reasoning traces that are not useful rebuild instructions

Use summarized reconstruction instructions instead.

The ledger should preserve **rebuildable intent**, not private scratchpad.

# 23. Quality Bar

The Reconstruction Ledger succeeds if future agents can answer:

* What were we trying to build?
* What instructions shaped it?
* What constraints mattered?
* What files and tools were involved?
* What should survive a rebuild?
* What should not be repeated?
* How can this be regenerated with a better model later?
* How do we verify the rebuilt artifact?
* Which decisions shaped the recipe?

It fails if:

* it becomes raw chat dump storage
* it omits constraints
* it omits known mistakes
* it cannot actually help rebuild anything
* it stores secrets
* it duplicates Decisions or Audit without adding rebuild value
* it is not updated after generation patterns change

This ledger should make the project recoverable, upgradeable, and less dependent on memory.

---

# 24. Final Operating Rule

The Reconstruction Ledger should end with this:

```markdown id="ccvwje"
The artifact is what exists.

The reconstruction ledger is how to make it exist again.

Preserve the recipe, not the noise.
```
