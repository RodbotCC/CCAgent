# File Contents Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **File Contents Ledger** explains what important files do.

The File Directory Ledger tells agents where things live.
The Directory Configuration Ledger tells agents how to behave inside important directories.
The File Contents Ledger tells agents what key files are responsible for.

This ledger is not meant to summarize every line of code.

It exists because many agent failures happen at the file level:

* editing a large shared file without understanding its role
* putting new logic into the wrong module
* changing a source file without updating the generated output pipeline
* editing a file that looks canonical but is actually derived
* missing the file that should have been changed instead
* failing to update related files after a change

The uploaded ledger plan describes the File Contents Ledger as practical orientation for important files: what the file does, when it should be edited, what should not be added, what depends on it, what it depends on, and what common mistakes happen around it. 

The goal is simple:

> A future agent should not have to blindly grep its way into understanding.

---

# 1. Core Operating Principle

The File Contents Ledger should enforce this rule:

> Important files should have a known job.

If a file is central to the system, agents should be able to quickly answer:

* What does this file do?
* Is this file source, generated, config, template, archive, or documentation?
* When should it be edited?
* When should it not be edited?
* What files does it depend on?
* What depends on it?
* What should be updated if it changes?
* What mistakes commonly happen around it?
* What verification should run after changing it?

This ledger is the “what is this file?” layer of the project memory system.

# 4. What This Ledger Owns

## Owns

* important file descriptions
* file type/status labels
* edit guidance
* dependency notes
* downstream impact notes
* verification notes
* “do not add here” warnings
* source/generated/template/archive labels
* file-level Done Gate triggers
* common mistake notes
* links to related ledgers/docs

## Does not own

* every file in the repo
* full code documentation
* line-by-line explanations
* all widget dependencies
* all API docs
* every historical edit
* all open bugs

Those belong in code comments, Page Ledgers, Widget Ledgers, API docs, Open Problems Ledger, Audit Ledger, and local Box files.

This ledger should focus on files that future agents are likely to touch or misunderstand.

# 5. Which Files Deserve Entries

Do **not** create entries for every file automatically.

Create File Contents Ledger entries for files that meet one or more of these conditions:

```markdown id="l4skp4"
A file deserves an entry if it:

- is edited often
- is large or crowded
- controls routing
- controls page rendering
- controls server/API behavior
- defines guardrails or policy
- defines source-of-truth rules
- generates output
- is generated output that agents might accidentally edit
- stores important configuration
- controls automation
- controls client state
- is a canonical template
- has known common mistakes
- connects multiple systems
- requires companion updates when changed
```

# 8. File Status Labels

Standardize file labels.

```markdown id="9fhhhd"
## 3. File Status Labels

Use one or more:

- `canonical`: source of truth
- `source`: human/agent-editable implementation source
- `generated`: derived output; do not hand-edit
- `config`: configuration or settings
- `policy`: rules/guardrails
- `template`: reusable template
- `runtime`: used by running system
- `script`: executable build/transform/check file
- `ledger`: memory/continuity file
- `archive`: historical reference
- `legacy`: old system, use cautiously
- `sensitive`: contains client, credential, or risky operational context
- `high-risk`: changes can create unsafe behavior
```

A file can have multiple labels.

Example:

```text id="1ysxw1"
Auto/comeketo-inbox/references/guardrails.md
status: canonical + policy + high-risk
```

# 19. Relationship to Page-Asset Sitemap

The File Contents Ledger should not duplicate every page section from the sitemap.

Instead:

* document major files like `app.jsx`, `screens.jsx`, `server.py`
* point page-specific ownership to the sitemap
* remind agents that page/route/data-binding edits require sitemap updates

The uploaded sitemap already maps pages, route index, primary screen components, APIs, side effects, and change checklists. 

The File Contents Ledger should say:

```markdown id="16hebu"
For page-specific asset ownership, the page-asset sitemap is authoritative.
This ledger only explains the high-level role of shared files.
```

# 20. Relationship to Source-of-Truth Ledger

The File Contents Ledger says what files do.

The Source-of-Truth Ledger says which file/system should be trusted for each kind of information.

Example:

```text id="rqwgho"
File Contents Ledger:
01_comms.md is conversation history.

Source-of-Truth Ledger:
For customer-facing allowed facts, trust full transcripts and 01_comms.md before profile/enrichment.
```

They should link to each other.

# 21. Relationship to File Directory Ledger

The File Directory Ledger maps folders.
The File Contents Ledger maps important files.

Do not merge them.

Example:

```text id="6lisuj"
File Directory Ledger:
Auto/orchestrator owns runtime/render/state layer.

File Contents Ledger:
Auto/orchestrator/bin/comms_state_sweep.py updates client comms/state and regenerates ledgers.
```

This separation keeps both readable.

# 23. Quality Bar

The File Contents Ledger succeeds if future agents can answer:

* What does this important file do?
* Should I edit it?
* What should I not put here?
* What depends on this file?
* What does this file depend on?
* What else must be updated if I change it?
* Is this file canonical or generated?
* What mistakes happen around this file?

It fails if:

* it tries to document every file
* it becomes line-by-line code commentary
* it is not updated when major file roles change
* it duplicates page/widget ledgers
* agents still edit generated files accidentally
* high-risk files lack warnings
* companion updates are missing

The File Contents Ledger should reduce the cost of safe editing.

---

# 24. Final Operating Rule

The File Contents Ledger should end with this:

```markdown id="04y3gt"
An important file should not be a black box.

Before editing it, know its job.
After changing it, update what depends on its job.
```
