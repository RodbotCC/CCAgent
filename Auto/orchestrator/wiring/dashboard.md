# dashboard.md — Pipeline Dashboard Wiring

`state/dashboard.html` is the rolled-up pipeline view. Generated by
`bin/refresh.py`, which is the implementation of CIA automations
**AUTO.10** (ledger-rollup) and **AUTO.11** (dashboard-refresh).

## Data pipeline

```
Client Boxes/<Lead Name>/    ──┐
                               │  bin/refresh.py
LEAD_SOURCES (in _lib.py)      │  ┌─ rollup_ledger()
  ├ alias / lead_id            ──┼─→  state/master_ledger.csv  (AUTO.10)
  ├ altitude / stage / day        │
                                  └─ compute_dashboard()
                                       ↓
                                state/dashboard.json  (AUTO.11)
                                       ↓
                                  render_dashboard_html()
                                       ↓
                                state/dashboard.html  (AUTO.11)
```

## File contracts

### Input: `Auto/Client Boxes/<Lead Name>/`

`enumerate_client_boxes()` walks the directory and classifies every
sub-folder as one of:

- **active** — has a matching alias in `LEAD_SOURCES` (right now: only
  `Hugo Casillas` → `hugo`)
- **intake-enriched** — folder contains `<Name>_enrichment.md`
- **intake** — folder exists but no enrichment file present yet

This is intentionally cheap: no filesystem reads beyond directory
listing. It scales to the full 28-lead substrate without slowing the
build.

### Output: `state/master_ledger.csv`

Schema:

| column | meaning |
|---|---|
| `lead_id` | Close lead ID; empty for non-active leads |
| `name` | folder name |
| `altitude` | whale / strong / standard / soft / — |
| `stage` | active / intake-enriched / intake |
| `files_present` | count of files in the lead's folder |
| `has_enrichment` | yes / no |
| `has_close_export` | yes / no |
| `current_plan_day` | 1–7 for active leads; empty otherwise |
| `current_off_ramp` | follow-up / close / hold / block / replan |
| `last_updated` | ISO timestamp of this rollup |

This CSV is the canonical machine-readable view. CIA §00.02. Future
runs of `refresh.py` overwrite it.

### Output: `state/dashboard.json`

Schema (top-level keys):

```json
{
  "generated_at": "<ISO>",
  "rollup": { "total": int, "active": int, "intake_enriched": int, "intake_raw": int, "altitude_counts": {...} },
  "active_leads": [ { "lead_id", "name", "altitude", "current_plan_day", "current_off_ramp", "last_updated" } ],
  "intake_enriched": [ name, name, … ],
  "intake_raw": [ name, name, … ],
  "andre_queue_24h": [ string, string, … ],
  "system_queue_24h": [ string, string, … ],
  "health_flags": [ string, string, … ]
}
```

Used by `bin/index.py` to populate the landing page numbers.

### Output: `state/dashboard.html`

Widgets, top to bottom:

| widget | reads | source key |
|---|---|---|
| Masthead | (constant) | NOW timestamp |
| KPI row | dashboard.json:rollup | `total` / `active` / `intake_enriched` / `intake_raw` |
| Active leads table | dashboard.json:active_leads | per-lead row, link to Lead Box |
| Andre + System queues | dashboard.json:andre_queue_24h / system_queue_24h | bullet lists |
| Intake queue | dashboard.json:intake_enriched / intake_raw | name pills |
| Pipeline health | dashboard.json:health_flags | amber-banded list |

## Discipline

- The dashboard **renders** state. It does not **modify** state. If
  `refresh.py` were a no-op, opening `dashboard.html` would still work
  (with the previous run's data).
- Per-lead state changes flow through `client_ledger.md` (still the
  per-lead source of truth). `refresh.py` reads enough to roll up; the
  Lead Box reads everything.
- **AUTO.10 ≠ AUTO.11**. They live in the same script for now because
  they're tiny, but they're separable: AUTO.10 writes the CSV; AUTO.11
  reads the CSV (transitively, via `compute_dashboard(rows)`) and emits
  JSON + HTML. If we ever want to roll up without rendering (or render
  without re-rolling), split them.

## Future plug-points

- **AUTO.10 cron** — `launchd` plist running `python3 bin/refresh.py`
  every 02:00. Or a file-watcher that fires `refresh.py` whenever any
  per-lead `client_ledger.md` is modified.
- **Multi-lead support** — when a second lead onboards, add an entry to
  `LEAD_SOURCES` in `_lib.py`. The rollup picks it up automatically.
- **Real `clients/<lead_id>/` migration** — `enumerate_client_boxes()`
  is the only place that knows about the current `Client Boxes/`
  directory. Update that function, dashboard adapts.
