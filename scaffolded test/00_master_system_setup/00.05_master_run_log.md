# 00.05 Master Run Log

## Deliverable

Create the master run log that records every pipeline run, timestamp, status, and failure.

## Purpose

The master run log is the chronological operating history for the entire Client Intelligence Agent system. It records each run or semi-run the system performs, what triggered it, which stage or lead it touched, whether it completed, and what failed.

The master ledger answers: “Where is each lead now?”

The master run log answers: “What happened, when, and why?”

## Recommended Location

```text
scaffolded test/logs/master_run_log.md
```

## Run Log Responsibilities

This log should track:

1. Every full pipeline run
2. Every partial stage run
3. Every manual/semi-automated scaffold action
4. Every automation-triggered action
5. Every lead-specific operation that changes state
6. Every failure, warning, skipped step, or blocked condition
7. Every retry and its result
8. Every GitHub commit or artifact write connected to a run
9. The operator or agent responsible for the action
10. The next expected action after the run completes

## Suggested Columns

| Column | Purpose |
|---|---|
| Run ID | Stable identifier for the run entry |
| Timestamp | When the run or action happened |
| Trigger | manual, scheduled, webhook, Close activity, GitHub commit, P-proceed, etc. |
| Scope | master, stage, lead, automation, dashboard, skill |
| Stage | Blueprint stage touched, if applicable |
| Deliverable ID | Atomic deliverable ID, if applicable |
| Lead Registry ID | Related lead, if applicable |
| Lead Name | Human-readable lead name, if applicable |
| Action Summary | Short description of what happened |
| Status | complete, in_progress, failed, blocked, skipped, needs_review |
| Failure Type | none, tool_error, missing_source, duplicate, permission, validation, unknown |
| Failure Detail | Short explanation if there was a failure |
| Artifact Path | File/folder created or updated |
| Commit SHA | GitHub commit SHA when available |
| Actor | Human or agent that performed the run |
| Next Action | Expected follow-up action |

## Run ID Convention

Use a simple ascending ID unless a more advanced automation later replaces it:

```text
run-000001
run-000002
run-000003
```

Lead-specific sub-runs may optionally extend this format:

```text
run-000104-lead-0007
```

## Logging Rules

1. Every completed atomic deliverable should create or imply a run log entry.
2. Every failed tool action should be logged with `failed` and a failure type.
3. Every intentionally skipped step should be logged as `skipped`, not silently omitted.
4. Every blocked state should include a next action.
5. Every GitHub write should include the resulting commit SHA when available.
6. The master run log should stay chronological.
7. Detailed per-lead logs may live in each client folder, but the master run log should preserve the system-level timeline.
