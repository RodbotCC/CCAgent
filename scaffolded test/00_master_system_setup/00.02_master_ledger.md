# 00.02 Master Ledger

## Deliverable

Create the master ledger that tracks every lead through every stage of the pipeline.

## Purpose

The master ledger is the system-wide source of truth for pipeline progress. It lets the operator, agent, or automation see where each lead is, what has already been completed, what is blocked, and what should happen next.

This ledger does not replace per-client ledgers. Instead, it gives one cross-lead view while each client folder keeps its own detailed local checklist.

## Recommended Location

```text
scaffolded test/master_ledgers/master_ledger.md
```

## Ledger Scope

The master ledger should track:

1. Lead identity
2. Close source information
3. Folder creation state
4. Raw comms retrieval state
5. People-search state
6. Deep-dive research state
7. Profile synthesis state
8. Seven-day plan state
9. If/then logic state
10. Skill assignment state
11. Automation state
12. Andre notification and alert state
13. Lead Liaison Agent readiness
14. Current live state
15. Outcome or next-state classification

## Status Vocabulary

Use consistent values so automations can read the ledger cleanly:

```text
not_started
in_progress
complete
blocked
needs_review
not_found
out_of_bounds
skipped
```

## Update Rules

1. Every newly harvested lead must be added to the master ledger.
2. Every stage completion should update the corresponding status field.
3. Every blocked state should include a short blocker note.
4. Per-lead ledger files may contain more detail, but the master ledger should always show the latest operational state.
5. The master ledger should avoid private speculation. It should track workflow status, not sensitive profile interpretation.
