# 00.04 Master Lead Registry

## Deliverable

Create the master registry that stores every lead discovered from Close, including lead name, pair name, Close lead ID, and Close URL.

## Purpose

The master lead registry is the canonical identity list for harvested leads. It separates lead identity from stage progress: the master ledger tracks pipeline state, while the master lead registry tracks who each lead is, where the lead came from, and how to locate the lead again.

This prevents duplicate lead folders, supports deduplication across multiple smart views, and gives downstream agents a stable lookup table for Close links and client folder paths.

## Recommended Location

```text
scaffolded test/registries/master_lead_registry.md
```

## Registry Responsibilities

This registry should track:

1. Lead display name from Close
2. Normalized folder-safe lead name
3. Pair/couple name, when applicable
4. Primary person name, when known
5. Secondary person name, when known
6. Close lead ID
7. Close lead URL
8. Source smart view label
9. Source smart view ID
10. Harvest order
11. Client folder path
12. Duplicate/merge status
13. Current ownership or assignment, when relevant
14. Date first discovered
15. Date last refreshed

## Suggested Columns

| Column | Purpose |
|---|---|
| Lead Registry ID | Internal stable identifier for this registry row |
| Lead Display Name | Name exactly as shown in Close |
| Normalized Lead Name | Folder-safe version of the lead name |
| Pair Name | Couple/family/household label when applicable |
| Primary Person | First known decision-maker or contact |
| Secondary Person | Partner/spouse/second decision-maker when applicable |
| Close Lead ID | Unique Close lead identifier |
| Close URL | Direct link to the Close lead record |
| Source Smart View Label | Human-readable source view name |
| Source Smart View ID | Unique Close smart view identifier |
| Harvest Order | Order found in the source smart view |
| Client Folder | Path to the lead's folder |
| Duplicate Status | unique, possible_duplicate, merged, ignored |
| Merged Into | Target lead registry ID if merged |
| Owner | Responsible human or agent, if assigned |
| First Discovered | Timestamp of first harvest |
| Last Refreshed | Timestamp of latest refresh |
| Notes | Identity or routing notes |

## Registry Rules

1. Every harvested Close lead must have exactly one active registry row unless it is marked `merged` or `ignored`.
2. Close lead ID is the strongest deduplication key when available.
3. Close URL is the second strongest deduplication key.
4. Lead display name alone should never be treated as a unique identifier.
5. Pair names should be used only when the lead is clearly a couple, household, or multi-person decision unit.
6. Client folder paths should be deterministic and should not change unless a rename is intentionally recorded.
7. If two rows are merged, preserve both source records and mark the losing row as `merged`.
8. The master ledger should reference this registry rather than duplicating all identity logic.
