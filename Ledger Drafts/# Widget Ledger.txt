# Widget Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Widget Ledger** tracks a specific component, panel, card, tool, renderer, control, or interaction surface when that widget is important enough to deserve its own memory.

The Page Ledger explains a whole page.

The Widget Ledger explains the meaningful pieces inside that page.

The uploaded ledger plan says a Widget Ledger should track what a widget does, where it renders, what owns it, what it reads, what it writes, what it can trigger, what settings affect it, what APIs/models/services it uses, what depends on it, and what done means for that widget. It should exist for widgets with data flow, API calls, side effects, model calls, state, settings, or dependencies. 

This is especially important in this project because some “widgets” are not harmless UI.

A widget might:

* save credentials
* submit a delegation
* trigger an automation
* render generated state as if it were truth
* rewrite a draft
* ingest files
* display client strategy
* approve a risky move
* send or prepare customer-facing copy

The Widget Ledger makes those surfaces safe to understand and change.

---

# 1. Core Operating Principle

The Widget Ledger should enforce this rule:

> A widget is not just what it looks like. A widget is what it reads, writes, trusts, triggers, and can break.

A future agent should be able to open a Widget Ledger and answer:

* What does this widget do?
* Where does it render?
* What page owns it?
* What file owns it?
* What data does it read?
* What data does it write?
* What APIs does it call?
* What side effects can it trigger?
* What source of truth does it trust?
* What settings affect it?
* What user states does it support?
* What can break if it changes?
* What must be updated when it changes?

A small-looking widget can be a large operational contract.

This ledger makes that contract explicit.

# 3. Role in the Ledger System

The Widget Ledger sits under Page Ledger and Asset / Widget Map.

```text
Asset / Widget Map
  └── global map of pages, widgets, APIs, data, side effects

Page Ledger
  └── page-specific purpose, states, source truth, APIs, side effects

Widget Ledger
  └── widget-specific behavior, reads, writes, state, risks, settings, Done Gate
```

Useful mental model:

```text
Page Ledger = room map
Widget Ledger = machinery label
```

Some widgets are just furniture.

Some widgets are machinery.

Only the machinery needs a ledger.

---

# 4. What This Ledger Owns

## Owns

* widget purpose
* render locations
* owning page
* owning file/component
* props/state/data contract
* API calls
* model/tool calls
* external service dependencies
* writes and side effects
* source-of-truth assumptions
* user states
* approval requirements
* guardrails
* settings dependencies
* known failure modes
* verification checklist
* widget-specific Done Gate
* dependency notes
* recent widget changes
* open widget problems

## Does not own

* entire page behavior
* every CSS detail
* every tiny presentational component
* global source-of-truth rules
* backend implementation details
* all page API documentation
* every reusable component

Those belong in Page Ledger, File Contents Ledger, Source-of-Truth Ledger, Asset / Widget Map, code comments, and API docs.

The Widget Ledger should focus on meaningful, stateful, risky, or confusing widgets.

---

# 5. Which Widgets Deserve Ledgers

Do **not** create a ledger for every tiny UI element.

Create a Widget Ledger when a widget:

```text
- reads important data
- writes data
- calls APIs
- triggers side effects
- submits approvals
- touches credentials
- interacts with external services
- renders generated or source-of-truth-sensitive data
- is shared across many pages
- has complex state
- has model/AI behavior
- affects customer-facing actions
- is frequently misunderstood
- has caused bugs before
```

High-priority candidates:

```text
Boxes dossier renderer
Automation workflow graph
Delegation approval controls
Settings credential editor
Intake ingest panel
Shared Markdown renderer
Risk approval card, once built
Allowed-to-know fact panel, once built
Scheduled-fire review card, once built
```

This is a selective ledger.

Quality matters more than coverage.

# 8. Widget Status Labels

```markdown
## 3. Widget Status Labels

Use one or more:

- `active`: currently used
- `shared`: used across multiple pages
- `read-only`: displays data only
- `write-capable`: can persist or mutate data
- `approval-capable`: can approve or submit actions
- `external-action`: can trigger an external service/channel
- `customer-facing-risk`: can affect customer-facing content/actions
- `credential-sensitive`: handles or displays credentials/settings
- `generated-view`: displays generated or cached state
- `source-editing`: edits canonical source data
- `high-risk`: changes can create serious side effects
- `needs-audit`: widget needs review
- `experimental`: not stable
- `legacy`: avoid new work unless audited
```

Examples:

```text
Boxes dossier renderer       active + generated-view + high-risk
Automation workflow graph    active + write-capable + high-risk
Delegation approval controls active + approval-capable + external-action + high-risk
Settings credential editor   active + credential-sensitive + write-capable + high-risk
Shared Markdown renderer     active + shared + high-risk
```

# 10. High-Risk Widget Index

Purpose: make dangerous widgets impossible to miss.

```markdown
## 5. High-Risk Widget Index

### Delegation approval controls

Risk:
Can submit or rewrite drafts and potentially trigger external target actions.

Needs:
Approval rules, target visibility, write logs, rollback/handoff clarity.

### Automation workflow graph

Risk:
Can change persisted automation behavior.

Needs:
Clear write targets, trigger/workflow validation, audit trail.

### Settings credential editor

Risk:
Can save or clear sensitive runtime credentials.

Needs:
Masking, allowlist, no raw secret exposure, verification checks.

### Boxes dossier renderer

Risk:
Can visually imply generated or derived data is canonical truth.

Needs:
Source/freshness labels, canonical/generated distinction.

### Shared Markdown renderer

Risk:

Cross-page verification after changes.

Risk:
Will control whether risky customer-facing moves get approved.

Needs:
Explicit risk class, exact approved wording/action, approval log, expiry/reuse rules.
```

# 11. Example Widget Record — Boxes Dossier Renderer

```markdown
# Widget Ledger — Boxes Dossier Renderer

Last updated: 2026-04-28
Widget ID: `boxes_dossier_renderer`
Widget name: Boxes dossier renderer
Status: active / generated-view / high-risk
Risk level: high
Renders on: `boxes`
Owning page: Boxes Page
Owner file: `screens.jsx`
Owner component: `BoxesScreen` or child component, verify actual name
Related APIs:
- `/api/boxes/list`
- `/api/boxes/<id>`
- `/api/boxes/<id>/html`
- `/api/boxes/<id>/template/<slug>`
Related source-of-truth entries:
- Client Boxes canonical truth
- Staff Boxes canonical truth
- Orchestrator generated state

## 1. Widget Purpose

Displays a selected Box dossier in the UI.

For Client Boxes, it should help the operator inspect the client’s memory/state package: comms, profile, plan, alerts, completeness, generated previews, and related handoff actions.

## 2. Render Locations

Primary render location:
- `boxes` route

Related page:
- `/LEDGERS/PAGES/boxes.md`

## 3. Ownership

Frontend owner:
- `screens.jsx`

Backend/API owner:
- `server.py` `/api/boxes/*` endpoints

Canonical data owner:
- `Auto/Client Boxes/<Name>/`
- `Auto/Staff Boxes/`

Generated data owner:
- `Auto/orchestrator/state`

## 4. Inputs / Reads

Reads:

- box list
- selected box metadata
- source file contents
- generated HTML preview
- template preview
- completeness/readiness information
- possibly delegation handoff metadata

## 5. Outputs / Writes

Default expected behavior:
- no direct writes to canonical Client Boxes

Possible write:
- optional delegation draft/handoff action, if enabled

If future editing is added:
- must be documented as source-editing and gated by Client Box source-of-truth rules.

## 6. APIs / Tools / Models

APIs:

- `/api/boxes/list`
- `/api/boxes/<id>`
- `/api/boxes/<id>/html`
- `/api/boxes/<id>/template/<slug>`

No model calls expected unless future summarization or rewrite actions are added.

## 7. Side Effects

Expected:
- read-only display

Possible:
- delegation draft creation through explicit action

Not allowed silently:
- rewriting client plans
- editing generated state
- sending customer-facing messages
- approving risky moves

## 8. Source-of-Truth Rules

The widget must not imply that UI display is canonical.

For client data:
- canonical source is `Auto/Client Boxes/<Name>/`

For generated preview:
- source is generated/rendered state, not canonical truth

If displayed data looks wrong:
1. check Client Box source files
2. check generator/orchestrator state
3. then check widget rendering

## 9. State Model

Supported states should include:

- loading list
- list loaded
- no boxes found
- selected box loading
- selected box loaded
- selected box missing files
- selected generated preview unavailable
- stale generated state warning
- API error
- delegation handoff pending/success/error, if handoff exists

## 10. User Actions

Possible actions:

- select a box
- view source sections
- view generated HTML/template preview
- trigger handoff/delegation, if present

Future possible actions:
- edit box file
- mark audit status
- approve next move

Any future write action must get a new Widget Ledger update.

## 11. Guardrails / Approval Rules

- Do not confuse generated state with source truth.
- Do not allow customer-facing actions without explicit approval.
- Do not surface enrichment-only facts as customer-facing allowed facts unless allowed-to-know layer supports it.
- Do not edit `client_ledger.md` as if it were source.

## 12. Settings / Configuration

Potential settings:

- selected box ID
- read-only/edit mode, if implemented
- generated preview enabled/disabled
- template preview enabled/disabled

Verify actual settings from code.

## 13. Dependencies

Depends on:

- `screens.jsx`
- `server.py`
- `/api/boxes/*`
- `Auto/Client Boxes`
- `Auto/Staff Boxes`
- `Auto/orchestrator/state`
- page-asset sitemap
- Asset / Widget Map

Depended on by:

- operator review flow
- Client Box audit workflow
- possible delegation workflow

## 14. Known Failure Modes

- Displays stale generated state without warning.
- Displays derived state as if canonical.
- Missing file renders as blank instead of explicit missing state.
- Handoff action writes unexpected target.
- UI bug causes operator to trust incomplete Client Box.

## 15. Widget Done Gate

A change to this widget is not done until:

- source/read/write behavior is documented
- page-asset sitemap updated if page/data binding changes
- Asset / Widget Map updated if dependencies changed
- Page Ledger `boxes.md` updated if page behavior changed
- Source-of-Truth Ledger updated if authority assumptions changed
- verification performed against at least one known box

## 16. Verification Checklist

- load boxes list
- select known Client Box
- compare displayed file sections to actual source files
- verify generated preview is labeled as generated
- verify missing source file state
- verify no write occurs during read-only navigation
- verify handoff action writes only expected target, if tested

## 17. Open Widget Problems

- Need clearer source/freshness labels.
- Need integration with allowed-to-know layer once implemented.
- Need verify actual component breakdown in repo.

## 18. Recent Widget Changes

Seed from sitemap/history when available.

## 19. Related Files / Ledgers

- `/LEDGERS/PAGES/boxes.md`
- `/LEDGERS/ASSET_WIDGET_MAP.md`
- `/LEDGERS/SOURCE_OF_TRUTH.md`
- `/LEDGERS/FILE_CONTENTS_LEDGER.md`
- page-asset sitemap
```

The existing sitemap says the `boxes` page reads `Auto/Client Boxes`, `Auto/Staff Boxes`, and `Auto/orchestrator/state`, and exposes `/api/boxes/*` APIs, so this widget must preserve the display-vs-source distinction.

```markdown
- [ ] verification performed or honestly marked unavailable
```

For high-risk widgets:

```markdown
High-risk widget changes require extra review.

Do not merge/change blindly if the widget can:
- write data
- approve actions
- touch credentials
- trigger external services
- influence customer-facing sends
- display generated state as truth
```

---

# 17. Widget State Standards

Every meaningful widget should define states.

```markdown
## 7. Widget State Standards

A widget should define whether it supports:

- loading
- loaded
- empty
- missing source
- stale data
- validation error
- API error
- permission error
- write pending
- write success
- write failure
- confirmation required
- approval required
- read-only mode
- disabled mode
```

High-risk widgets need explicit failure states.

A dangerous widget should not fail silently.

---

# 18. Widget Verification Standards

Default verification:

```markdown
## 8. Widget Verification Standards

When changing a widget, verify:

- it renders in every expected location
- expected data appears
- empty state works
- error state works
- write action goes to expected target
- no unexpected side effect occurs
- source-of-truth assumptions remain valid
- guardrails/approval rules still apply
- related pages still work
```

For shared widgets:

```markdown
Verify at least two independent usage locations.
```

For credential widgets:

```markdown
Verify no raw secret is exposed.
```

For approval widgets:

```markdown
Verify risky action cannot be approved accidentally.
```

# 21. Relationship to Page Ledger

A Page Ledger should list widgets.

A Widget Ledger should explain the complex widgets.

Example:

```text
Page Ledger — boxes:
Lists dossier renderer, template preview, handoff action.

Widget Ledger — boxes_dossier_renderer:
Explains exact reads, source truth, state, side effects, verification.
```

Do not put every widget detail inside the Page Ledger.

---

# 22. Relationship to Asset / Widget Map

The Asset / Widget Map is the global registry.

The Widget Ledger is the detailed record.

```text
Asset / Widget Map:
BoxesScreen uses Boxes dossier renderer.

Widget Ledger:
Boxes dossier renderer reads Client Boxes, generated orchestrator state, supports stale/missing states, and must not imply generated truth.
```

The Asset / Widget Map should link to widget ledgers.

---

# 23. Relationship to Source-of-Truth Ledger

Widgets need local source-of-truth rules.

The Source-of-Truth Ledger owns the global rules.

Example:

```text
Widget:
Boxes dossier renderer displays `client_ledger.md`.

Source-of-Truth Ledger:
`client_ledger.md` is generated/derived; canonical comms live in `01_comms.md` and transcripts.
```

If a widget starts editing canonical data, both ledgers need updates.

# 24. Relationship to Settings Ledger

Some widgets are settings-dependent.

Example:

```text

it is itself part of Settings Ledger.

Automation workflow graph:
may depend on model/provider settings.

Shared Markdown renderer:
may depend on markdown/sanitizer options if configurable.
```

If a setting affects widget behavior, document it.

# 25. Relationship to Connections Ledger

If a widget introduces a new external dependency, update Connections.

# 27. Widget Change Done Gate

Update this ledger when:

```markdown
- a high-risk widget is created
- a widget starts reading new data
- a widget starts writing data
- a widget starts calling a new API/model/tool
- a widget triggers a new side effect
- a widget source-of-truth assumption changes
- a widget’s approval behavior changes
- a widget becomes shared across pages
- a shared widget’s rendering/sanitization behavior changes
- a widget is audited
- a widget problem is discovered or resolved
```

Do not update for tiny style-only changes unless they alter state, behavior, side effects, or user expectation.

# 29. Quality Bar

The Widget Ledger succeeds if future agents can answer:

* What does this widget do?
* Where does it render?
* What owns it?
* What does it read?
* What does it write?
* What APIs/tools/models does it use?
* What source truth does it trust?
* What side effects can it trigger?
* What settings affect it?
* What states does it support?
* What could break if it changes?
* What does done mean for this widget?

It fails if:

* every tiny UI element gets a bloated ledger
* high-risk widgets lack records
* side effects are not documented
* source-of-truth assumptions are missing
* shared widgets are changed without cross-page verification
* approval/credential widgets are treated like normal UI
* widget ledgers duplicate page ledgers instead of deepening them

This ledger should make risky widgets safe to change.

---

# 30. Final Operating Rule

The Widget Ledger should end with this:

```markdown
A widget is not just a component.

It is a behavior surface.

Before changing it, know what it reads.
Before trusting it, know what it writes.
Before approving it, know what it can trigger.
```
