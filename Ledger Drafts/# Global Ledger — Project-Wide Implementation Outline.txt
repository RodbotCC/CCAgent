

# Global Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Global Ledger** is the top-level continuity layer for the entire project.

It is the project’s durable “world state.”

It should tell any future agent, human, model, MCP server, or automation what this project is, how it is organized, what currently matters, what rules govern the work, where truth lives, what is active, what is risky, and what must be updated when the system changes.

The Global Ledger is not just documentation.

It is the project’s memory spine.

The local file tree is the working copy.
GitHub is the source of truth.
The repo is not just code. The repo is the memory of the build. 

The Global Ledger should make that principle real.

---

# 1. Core Operating Principle

The Global Ledger should enforce this rule across the project:

> If it matters to future work, it belongs in the versioned file tree.

That includes:

* code
* configuration
* ledgers
* maps
* decisions
* open problems
* page ownership
* source-of-truth rules
* automation rules
* human preferences
* current project state
* rebuild instructions
* visual maps
* audit notes
* handoff notes
* implementation history

The Global Ledger should be the first file an agent reads before doing meaningful work.

It should answer:

* What is this project?
* What is true right now?
* What are the major systems?
* What is currently active?
* What should agents know before touching anything?
* What are the global rules?
* What has changed recently?
* What needs to be updated when work happens?

The user’s ledger philosophy already defines the Global Ledger as the “top-level memory of the project” and the source of durable world state.


# 4. Global Ledger Responsibilities

The Global Ledger should own project-wide orientation, not local detail.

It should not become a dumping ground for everything.

It should point to local ledgers, not replace them.

## Owns

* project identity
* current global state
* major systems map
* root source-of-truth rules
* active phase
* active risks
* global done gates
* global update rules
* links to local ledgers
* handoff instructions for future agents
* current “read first” order
* global rules that apply everywhere

## Does not own

* detailed page implementation
* every widget dependency
* every local open problem
* every file summary
* every client-specific plan
* every API detail
* every code comment

Those belong in local ledgers, page ledgers, widget ledgers, directory configs, source-of-truth ledgers, and boxes.

The Global Ledger is the map of the world.
The local ledgers are the maps of neighborhoods.

Purpose: explain the project in one readable page.

Should answer:

* What is this project?
* Who is it for?
* What is it trying to become?
* What makes it different?
* What is the core design philosophy?

Example tone:

```markdown
## 1. Project Identity

This project is a GitHub-backed AI operations system for Comeketo. It combines client memory, automation, inbox work, page surfaces, ledgers, boxes, and agent handoff systems into a durable file-tree orchestration environment.
```

```markdown
The plan is not truth. The box is truth. The plan is a draft strategy subordinate to comms, state, guardrails, approvals, and source-of-truth rules.
```

## 6.5 Source-of-Truth Rules

Purpose: prevent agents from editing the wrong thing.

This is one of the most important Global Ledger sections.

Example:

```markdown
## 4. Source-of-Truth Rules

### 4.1 GitHub

GitHub is the durable source of truth for project memory, ledgers, code, configs, and handoff state.

### 4.2 Local file tree

The local file tree is the working copy. It may contain unpushed work. Agents must be careful when writing directly to GitHub if local changes may exist.

### 4.3 Client Boxes

For client-specific truth, trust:

1. `01_comms.md`
2. full transcripts when available
3. `00_meta.json`
4. `client_ledger.md`
5. approved operator notes
6. profile/enrichment only as internal strategy, not customer-facing truth

### 4.4 Generated outputs

Generated files are views, not primary truth, unless explicitly marked canonical.

### 4.5 Plans

Seven-day plans are strategy drafts. They do not override guardrails, current state, replies, approvals, or comms.

### 4.6 UI pages

UI pages render truth. They do not define truth unless explicitly stated in a source-of-truth ledger.
```

This section should connect directly to the future Source-of-Truth Ledger, but it should keep the global version concise.

## 6.6 Global Operating Rules

Purpose: define rules that apply everywhere.

Example:

```markdown
## 5. Global Operating Rules

1. Read before writing.
2. Prefer narrow, reversible changes.
3. Do not treat generated state as canonical unless marked.
4. Do not let plans override guardrails.
5. Risky automation moves require explicit approval.
6. If a client replies, their plan becomes stale until reviewed.
7. If a page/data binding changes, update the page-asset sitemap.
8. If project state changes, update the relevant ledger.
9. If a local directory has a Box, read it before editing there.
10. If a problem is found but not fixed, record it.
11. If a decision is made, record it.
12. If a change would confuse a future agent, leave a handoff note.
```

This is where the project’s “common law” lives.

## 6.8 Active Risks / Known Fragile Areas

Purpose: make risks impossible to miss.

Example:

```markdown
## 7. Active Risks / Known Fragile Areas

### Automation approval risk

Batch approval can hide high-risk moves. Fee waivers, discounts, scope promises, enrichment-based personalization, and pricing claims need isolated approval.

### Plan / guardrail contradiction

Some seven-day plans were written before guardrails were fully enforced. Plans may contain unsafe executable copy.

### Reply invalidates plan

Any meaningful inbound from a client can invalidate the existing seven-day plan.

### Enrichment boundary

Profiles may contain useful internal strategy, but customer-facing copy must only use facts from comms or explicit approval.

### GitHub/local drift

Agents may write directly to GitHub while the human has unpushed local changes. Always consider branch/pull/rebase safety.
```

This section should be aggressively honest.

If it is scary, write it down.

## 6.11 Agent Work Protocol

Purpose: give agents a repeatable workflow.

Example:

```markdown
## 10. Agent Work Protocol

Before meaningful edits:

1. Read `GLOBAL_LEDGER.md`.
2. Identify the affected system.
3. Read the relevant Box or directory orientation.
4. Read source-of-truth rules for the data being changed.
5. Check open problems and recent audit notes.
6. Inspect current files.
7. Make the smallest safe change.
8. Run available checks or explain why not.
9. Update affected ledgers.
10. Leave a handoff note if future agents need context.
11. Commit with a clear message.
12. Push to GitHub unless instructed otherwise.
```

For risky areas:

```markdown
If changing automation, sends, client plans, approval gates, source-of-truth bindings, or generated state, stop and check guardrails before writing.
```

This is the meta harness in human-readable form.

## 6.12 Done Gate

Purpose: make “done” include memory.

Example:

```markdown
## 11. Done Gate

Work is not complete until the memory of the work is updated.

Meaningful changes must update the relevant continuity surface.

Examples:

- Changed project state → update Global or Temporal Ledger.
- Changed a page → update page-asset sitemap / Page Ledger.
- Changed folder ownership → update Directory Ledger.
- Changed client plan → update client audit marker or local Box ledger.
- Changed source-of-truth rule → update Source-of-Truth Ledger.
- Made a major decision → update Decisions Ledger.
- Found but did not fix a problem → update Open Problems Ledger.
- Changed a repeatable process → update Reconstruction Ledger.
- Completed an audit → update Audit Ledger.
```

This connects directly to the user’s “Done Gate” idea from the uploaded ledger list. 

The short version:

```markdown
If the system changed, the system’s memory changes with it.
```

# 9. Relationship to Boxes

The Global Ledger should define Boxes as a core project primitive.

A Box is not just a folder.
A Box is a folder with memory, constraints, ledgers, local rules, source-of-truth notes, and handoff context.

Global Ledger owns the Box concept.
Each Box owns its local truth.

Example:

```text
Global Ledger
  └── defines Box system

Client Boxes
  └── each client has local state, comms, plan, guardrails, audit markers

Orchestrator Box
  └── owns runtime/rendering state rules

UI/App Box
  └── owns route/page/component/data-binding rules
```

This prevents global overreach.

The Global Ledger should say:

```markdown
Do not solve local complexity globally if it belongs inside a Box.
```

That line is extremely important.

```markdown
Automation may create movement, but it must not silently assume authority.

Safe moves:

Risk moves:

- urgency pressure
- enrichment-based personalization
- claims not found in comms

Risk moves must become approval cards, not automatic sends.
```

```markdown
If a lead replies, the plan is stale until reviewed.
```

And:

```markdown
The seven-day plan is a strategy draft, not a send license.
```

* what truth to trust
* what not to touch casually
* what risks are active

It fails if:

* it becomes a stale manifesto
* it duplicates every local doc
* it hides current risks
* it is too long to read
* it does not tell agents what to do
