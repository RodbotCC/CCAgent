# Asset / Widget Map Ledger â€” Project-Wide Implementation Outline

## 0. Purpose

The **Asset / Widget Map Ledger** explains how visible product surfaces connect to the underlying system.

It maps pages, widgets, panels, routes, APIs, stores, models, files, side effects, generated state, and source-of-truth dependencies.

This is one of the most important ledgers for UI-heavy builds.

The uploaded ledger plan says this ledger should track pages, widgets, panels, routes, APIs, servers, models, tools, stores, sources of truth, side effects, and dependencies. It should answer what renders where, what data each surface reads or writes, what breaks if it changes, and what other widgets depend on it. 

The project already has an early version of this idea in the **Comeketo Page-Asset Source Of Truth** file. That file maps route names, primary screen components, assets on each page, API ownership, side effects, change checklists, and history. It also defines a mandatory Done Gate for page, route, and page data-binding changes. 

The Asset / Widget Map Ledger should not replace that sitemap.

It should **promote it into a richer, project-wide dependency system**.

---

# 1. Core Operating Principle

The Asset / Widget Map Ledger should enforce this rule:

> A visible surface is not just UI. It is a contract between render, data, state, source truth, and side effects.

A widget is dangerous when no one knows:

* where it renders
* what owns it
* what data it reads
* what data it writes
* what API it calls
* what state it mutates
* what source of truth it trusts
* what side effects it can trigger
* what breaks if it changes

This ledger makes visible surfaces traceable.

The goal is not to create pretty diagrams.

The goal is to prevent hidden dependency bugs.

# 4. What This Ledger Owns

## Owns

* route-to-page map
* page-to-component map
* page-to-widget map
* widget ownership
* widget data dependencies
* widget write/side-effect dependencies
* page API dependencies
* source-of-truth links
* generated vs canonical data notes
* page/widget risk notes
* cross-widget dependencies
* visual dependency graphs
* mandatory update rules for visible assets

## Does not own

* every tiny DOM element
* every CSS class
* all implementation details
* full source-of-truth policy
* detailed file descriptions
* all API endpoint documentation
* every local page bug
* full task tracking

Those belong in code, styles, File Contents Ledger, Source-of-Truth Ledger, Page Ledgers, Widget Ledgers, and Open Problems Ledger.

The Asset / Widget Map Ledger should capture **meaningful product surfaces**, not every div.

# 13. Generated vs Canonical Surface Rules

Purpose: prevent UI from becoming false authority.

Example:

```markdown id="ew70q5"
## 9. Generated vs Canonical Surface Rules

UI surfaces may display generated or cached data.

Generated display is not automatically source truth.

Rules:

1. If a page shows Client Box data, canonical truth remains in `Auto/Client Boxes`.
2. If a page shows orchestrator state, check whether the state is generated.
3. If a page shows analytics snapshots, identify the snapshot generator/source.
4. If a page lets users write data, record where the write lands.
5. If a page changes source-of-truth ownership, update Source-of-Truth Ledger.
```

This should be repeated in relevant Page Ledgers.

# 15. Change Done Gate

Purpose: define when this ledger/sitemap must update.

Example:

```markdown id="933ljg"
## 11. Change Done Gate

Update the Asset / Widget Map or page-asset sitemap when:

- a route is added, removed, or renamed
- a page changes primary component
- a page changes data source
- a page changes API usage
- a page gains or loses side effects
- a widget starts reading new data
- a widget starts writing data
- a widget changes source-of-truth assumptions
- a shared widget changes behavior across pages
- a high-risk widget changes approval/write behavior
```

For page changes, the existing sitemap rule is mandatory: page, route behavior, or data-binding changes are incomplete until the sitemap is updated.

Do not overload the global map with every page nuance.

* frequently edited

# 20. Relationship to Source-of-Truth Ledger

This ledger should not become the full source-of-truth ledger.

It should point each surface to its canonical source.

The Source-of-Truth Ledger will define the authority rules.

Example:

```text id="xligem"
Asset / Widget Map:
BoxesScreen reads Auto/Client Boxes through /api/boxes/*.

Source-of-Truth Ledger:
For client plan truth, trust Auto/Client Boxes/<Name>/05_seven_day_plan.md only after comms/state/guardrail audit.
```

Both are needed.

One maps surface dependencies.
The other maps truth authority.

---

# 21. Relationship to Done Gate

The Asset / Widget Map should be wired into Definition of Done.

For any meaningful page/widget change:

```markdown id="bh9az6"
Done requires:
- page-asset sitemap updated
- Asset / Widget Map updated if dependencies changed
- Page Ledger updated if page behavior changed
- Widget Ledger updated if high-risk widget behavior changed
- Source-of-Truth Ledger updated if data authority changed
```

This keeps UI changes from becoming invisible architecture drift.

# 23. Quality Bar

The Asset / Widget Map Ledger succeeds if future agents can answer:

* What pages/routes exist?
* What component renders each page?
* What widgets live on each page?
* What data does each page/widget read?
* What does each page/widget write?
* What APIs does each surface call?
* What side effects can the surface trigger?
* What is the canonical source behind displayed data?
* What breaks if this widget changes?
* Which widgets are high-risk?
* What map or sitemap must be updated after changes?

It fails if:

* it becomes stale after page changes
* it duplicates the sitemap without adding dependency clarity
* it maps visuals but not data/side effects
* it ignores source-of-truth distinctions
* it tracks too many tiny components
* it does not flag high-risk widgets
* agents still change UI behavior without updating maps
