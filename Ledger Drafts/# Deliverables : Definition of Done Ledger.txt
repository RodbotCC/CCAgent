# Deliverables / Definition of Done Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Deliverables / Definition of Done Ledger** defines what “done” means across the project.

It protects the project from the most common false finish:

> “The code changed, so the work is done.”

In this build style, that is not enough.

A feature may not be done until the source-of-truth files are updated.
A page may not be done until the page-asset sitemap is updated.
A Client Box may not be done until the plan, guardrails, state, and audit marker are aligned.
A workflow may not be done until the relevant ledgers, visual maps, and handoff notes are updated.

The uploaded ledger plan says this ledger can exist globally, by phase, by page, by widget, by feature, or anywhere that needs a clear completion standard. It should answer what must exist before a phase, page, widget, or integration is complete, and what ledgers, maps, visualizations, checks, and cleanup are required. 

This ledger turns the project’s **55 / 45 build rhythm** into an actual operating system:

* 55% offense: build the thing.
* 45% defense: make sure the thing survives.

---

# 1. Core Operating Principle

The Definition of Done Ledger should enforce this rule:

> Done means the system changed and the system’s memory changed with it.

The project should not accept “done” as a vibe.

Done should be checkable.

Not every tiny edit needs every ledger updated. But every meaningful change should leave the relevant continuity surface current enough that future agents can understand, verify, rebuild, or safely continue the work.

The short rule:

```text id="c3jg9f"
If it changed behavior, ownership, truth, workflow, risk, or future context, it needs a Done Gate.
```

  └── What outcomes matter?

  └── Did completed work actually meet the Done Gate?

  └── What is done for this milestone?

  └── What is done for this specific surface?

It should be used by:

* agents before closing a task
* audits
* PR reviews
* page work
* widget work

* Client Box cleanup
* ledger creation
* refactors

# 4. What This Ledger Owns

## Owns

* global definition of done
* category-specific Done Gates
* completion checklists
* required ledger updates
* required sitemap updates
* required audit markers
* verification expectations
* cleanup expectations
* handoff expectations
* 55 / 45 build rhythm
* rules for when a task is not actually done

## Does not own

* all current tasks
* all open problems
* all phase deliverables
* page-specific asset details
* every test command
* every client-specific audit detail

Those belong in Phase Ledgers, Open Problems Ledgers, Page Ledgers, Widget Ledgers, local Box ledgers, and audit markers.

This ledger defines the **standard**.
Local ledgers apply the standard.

## 6.2 Core Definition of Done

Purpose: define the universal standard.

Example:

```markdown id="bkskoy"
## 1. Core Definition of Done

A meaningful task is done only when:

1. The intended change has been made.
2. The correct source of truth was changed, not merely a generated surface.
3. The affected system still works or the remaining risk is clearly recorded.
4. Relevant ledgers, maps, or local Box files are updated.
5. Any changed page, route, data binding, or visible asset has its sitemap/page ledger updated.
6. Any new risk, open problem, or unresolved follow-up is recorded.
7. Any future agent needing context can find it in the repo.
8. The change is committed to GitHub, unless intentionally held locally.
```

Short version:

```markdown id="xa8pwi"
Build the thing. Verify the thing. Record the thing. Hand off the thing.
```

# 7. 55 / 45 Build Rhythm

Purpose: make the user’s defense/offense rhythm actionable.

Example:

```markdown id="o052me"
## 2. 55 / 45 Build Rhythm

The project follows a 55 / 45 build rhythm.

### 55% Offense

Offense means creating or changing:

- features
- pages
- widgets
- APIs
- automations
- workflows
- client plans
- renderers
- integrations
- scripts
- models
- data pipelines

### 45% Defense

Defense means preserving the system:

- updating ledgers
- updating source-of-truth maps
- cleaning stale assumptions
- verifying routes and APIs
- writing audit markers
- recording decisions
- logging open problems
- updating visual maps
- adding handoff notes
- keeping GitHub current
- reducing future confusion

A build that remembers itself compounds.
A build that does not remember itself rots.
```

This section should give permission to spend real time on defensive work.

# 8. Universal Done Gate

Purpose: one global checklist for all meaningful tasks.

Example:

```markdown id="wval5m"
## 3. Universal Done Gate

Before calling work done, answer:

- [ ] Did I change the correct source of truth?
- [ ] Did I avoid editing generated output as canonical truth?
- [ ] Did I check whether any guardrail or source rule applies?
- [ ] Did I update any affected ledger, map, sitemap, or Box note?
- [ ] Did I record unresolved problems?
- [ ] Did I leave enough context for the next agent?
- [ ] Did I run or describe verification?
- [ ] Did I commit/push or clearly state why not?
```

Add a stronger version for agents:

```markdown id="slmfpn"
If any answer is “no,” the work may still be useful, but it is not fully done.
```

## 9.1 Client Box Done Gate

```markdown id="zyu4k1"
### Client Box Work

A Client Box pass is done when:

- [ ] Required files were checked:
  - `00_meta.json`
  - `01_comms.md`
  - `04_profile.md`
  - `05_seven_day_plan.md`
  - `09_andre_alerts.md`
  - `client_ledger.md` if present
- [ ] Current state was checked against the plan.
- [ ] Calendar labels and dates were checked.
- [ ] Reply Gate was checked.
- [ ] Frequency Cap risk was checked.
- [ ] Commitment language was checked.
- [ ] Enrichment boundary was checked.
- [ ] Customer-facing facts were checked against comms/transcripts.
- [ ] Future sends were flattened or approval-gated if risky.
- [ ] Any edits were narrow and local to that box.
- [ ] An audit marker was added or updated.
- [ ] The next recommended action is clear.
```

This is directly useful for the ongoing Client Box cleanup workflow.

---

## 9.2 Automation / Scheduled Fire Done Gate

```markdown id="eiwugy"
### Automation / Scheduled Fire Work

Automation work is done when:

- [ ] The trigger source is clear.
- [ ] The source of truth for input data is clear.
- [ ] Guardrails are applied before output.
- [ ] Reply/state changes can pause or invalidate the automation.
- [ ] Risky moves require explicit approval.
- [ ] Logs or run summaries are written.
- [ ] Failure mode is recorded.
- [ ] Human review path is clear.
- [ ] The automation does not silently make financial, pricing, scope, or enrichment-based commitments.
- [ ] Relevant ledgers or local Box instructions are updated.
```

---

## 9.3 Page / Route Done Gate

This should explicitly integrate the uploaded page-asset sitemap.

```markdown id="0zqb52"
### Page / Route Work

A page, route, or page data-binding change is done when:

- [ ] Route still exists and maps to the expected component.
- [ ] Primary component ownership is clear.
- [ ] Assets on page are listed or updated.
- [ ] Data sources are listed or updated.
- [ ] APIs are listed or updated.
- [ ] Side effects are listed or updated.
- [ ] Styles/cache-busters are updated if needed.
- [ ] The page-asset sitemap is updated.
- [ ] Last Verified date is bumped.
- [ ] History line is appended.
```

This matches the existing sitemap Done Gate: page, route behavior, and page data binding changes are incomplete until the sitemap is updated. That uploaded source explicitly says the relevant page section’s Asset Ownership, Change Checklist, History, and Last Verified must be updated.

## 9.4 Widget / Component Done Gate

```markdown id="u8ddck"
### Widget / Component Work

Widget work is done when:

- [ ] Widget purpose is clear.
- [ ] Render owner is clear.
- [ ] Inputs are clear.
- [ ] Outputs are clear.
- [ ] APIs or model calls are clear.
- [ ] Side effects are clear.
- [ ] Settings that affect it are documented.
- [ ] Dependent pages/widgets are checked.
- [ ] Asset/widget map is updated if the widget is meaningful.
- [ ] Visual or state regressions were checked where possible.
```

## 9.5 Source-of-Truth Change Done Gate

```markdown id="1xn9cp"
### Source-of-Truth Change

A source-of-truth change is done when:

- [ ] The old source is identified.
- [ ] The new source is identified.
- [ ] Generated/cached surfaces are labeled.
- [ ] Agents know what to trust first.
- [ ] Any affected scripts/renderers/API endpoints are updated.
- [ ] Source-of-Truth Ledger is updated.
- [ ] Page/widget ledgers or sitemap are updated if UI is affected.
- [ ] Migration risk is recorded.
```

## 9.6 Ledger Creation / Ledger Edit Done Gate

```markdown id="fcrvop"
### Ledger Work

Ledger work is done when:

- [ ] The ledger’s purpose is clear.
- [ ] Ownership is clear.
- [ ] Update rules are defined.
- [ ] Relationship to other ledgers is clear.
- [ ] Markdown exists for humans/agents.
- [ ] JSON exists if machine-readability is useful.
- [ ] Visual map exists if relationships are complex.
- [ ] The Global Ledger or Index links to it.
- [ ] The ledger avoids duplicating another ledger’s job.
```

## 9.7 Directory / Box Done Gate

```markdown id="6c861p"
### Directory / Box Work

A directory or Box change is done when:

- [ ] Directory purpose is clear.
- [ ] What belongs there is clear.
- [ ] What does not belong there is clear.
- [ ] Local source-of-truth rules are clear.
- [ ] Important files are listed.
- [ ] Local risks or guardrails are recorded.
- [ ] Local ledgers or orientation files are updated.
- [ ] Global Directory Ledger is updated if the file tree shape changed.
```

## 9.8 External Connection / API Done Gate

```markdown id="5hshna"
### External Connection / API Work

Connection work is done when:

- [ ] Service name and purpose are recorded.
- [ ] Required credentials are known without exposing secrets.
- [ ] Dependency owner is clear.
- [ ] Failure mode is understood.
- [ ] Billing/trial/renewal risk is recorded if relevant.
- [ ] What breaks if it fails is recorded.
- [ ] Connections Ledger is updated.
- [ ] Settings Ledger is updated if user-facing configuration changed.
```

---

## 9.9 Decision Done Gate

```markdown id="egj5yk"
### Decision Work

A major decision is done when:

- [ ] The decision is written down.
- [ ] Why it was made is recorded.
- [ ] Alternatives are briefly noted if useful.
- [ ] Consequences are recorded.
- [ ] Status is clear:
  - final
  - experimental
  - under review
  - deprecated
- [ ] Affected systems are listed.
- [ ] Decisions Ledger is updated.
```

---

## 9.10 Open Problem Done Gate

```markdown id="hsb42v"
### Problem Discovery

Finding a problem is done when:

- [ ] The problem is recorded.
- [ ] Severity or urgency is estimated.
- [ ] Affected systems are listed.
- [ ] Whether it was fixed, partially fixed, or left open is clear.
- [ ] Next recommended action is clear.
- [ ] Open Problems Ledger or local Box problem section is updated.
```

This is important because finding a problem but failing to record it forces future agents to rediscover it.

---

## 9.11 Reconstruction / Prompt Done Gate

```markdown id="u1ofx3"
### Rebuild / Prompt / Generation Work

A generated or rebuildable system is done when:

- [ ] Intent is recorded.
- [ ] Prompt/instructions are preserved if useful.
- [ ] Constraints are recorded.
- [ ] Source files and output files are listed.
- [ ] Regeneration path is clear.
- [ ] Known mistakes to avoid are recorded.
- [ ] Prompt / Reconstruction Ledger is updated.
```

# 11. Verification Requirements

Purpose: define what “verified” means.

Example:

```markdown id="3pqefq"
## 6. Verification Requirements

Verification can include:

- running tests
- running build scripts
- opening the relevant page
- checking generated output
- checking JSON validity
- checking Markdown links
- reviewing source-of-truth files
- checking route/component wiring
- checking client plan against comms
- checking guardrails against executable copy

If verification is not possible, record why.

Do not pretend verification happened.
```

```markdown id="zrhx9b"
An honest “not verified because…” is better than a fake green check.
```

# 13. What Does Not Count as Done

Purpose: make false completion visible.

Example:

```markdown id="c8ip2g"
## 8. What Does Not Count as Done

The following are not enough by themselves:

- Code was edited.
- A page looks okay once.
- A generated file changed.
- A plan was written but not checked against guardrails.
- A client box has files but no state alignment.
- A risky send was approved inside a batch without explicit risk surfacing.
- A decision was made in chat but not recorded.
- A bug was found but not logged.
- A source-of-truth change was made but maps still point to the old source.
- A local change exists but was not pushed or clearly marked local-only.
```

This section should be blunt.

# 14. Local Done Gates

Purpose: explain that local areas can define stricter done.

Example:

```markdown id="lgndoc"
## 9. Local Done Gates

The global Done Gate is the floor.

Local systems may define stricter gates.

Examples:

- Page-asset sitemap defines page-specific done.
- Inbox Guardrails define send safety done.
- Client Boxes define per-client plan/state done.
- Orchestrator may define generated-state done.
- Ledger Box may define memory-system done.
- External connection ledgers may define credential/dependency done.

When local and global Done Gates conflict, use the stricter requirement.
```

This allows precision without bloating the global file.

- What pauses it?
- What requires approval?
- What facts may it use?

- What happens after reply?

# 23. Quality Bar

The Definition of Done Ledger succeeds if it makes completion legible.

A future agent should be able to read it and know:

* whether work is actually done
* what checks apply
* what ledgers must be updated
* what verification is expected
* what handoff is needed
* what local Done Gates may override the global standard
* when “code changed” is not enough

It fails if:

* it becomes too vague
* agents ignore it before closing tasks
* it does not connect to page/client/automation realities
* it duplicates every local checklist
* it becomes a stale ceremony
* it makes tiny changes too heavy
* it lets meaningful changes close without memory updates

This ledger should create discipline without creating paralysis.

---

# 24. Final Operating Rule

The Definition of Done Ledger should end with this:

```markdown id="3mnu2t"
Done is not when the model stops writing.

Done is when the system can survive the change.

Build the thing.
Verify the thing.
Record the thing.
Hand off the thing.
```
