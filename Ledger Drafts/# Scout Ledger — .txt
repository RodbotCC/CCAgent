# Scout Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Scout Ledger** is the project’s disciplined exploration layer.

It tracks research into better tools, models, APIs, frameworks, services, patterns, automations, workflows, libraries, hosting options, and implementation approaches.

The uploaded ledger plan describes the Scout Ledger as the place to investigate better tools, models, APIs, frameworks, services, patterns, and workflows without randomly switching tools every time something shiny appears. It should record what tools are being considered, what problem they solve, what was tested, what worked, what failed, and what should be adopted, rejected, watched, or revisited. 

That is the key:

> The goal is not tool-chasing.
> The goal is disciplined exploration.

Modern AI projects are especially vulnerable to shiny-object drift.

The Scout Ledger gives curiosity a container.

---

# Scout Ledger — Project-Wide Implementation Outline

## 1. Core Operating Principle

The Scout Ledger should enforce this rule:

> Explore deliberately. Adopt cautiously. Remember what was already evaluated.

A project like this will constantly encounter new options:

* a better model
* a new MCP server
* a new dashboard framework
* a new browser automation tool
* a new vector/database/search option
* a new email renderer
* a new CRM API pattern
* a new local agent runtime
* a new GitHub automation workflow
* a new transcript processor
* a new visualization library

Without a Scout Ledger, the project will keep researching the same things repeatedly or switching tools without memory.

The Scout Ledger should help answer:

* What problem are we trying to solve?
* What options are we considering?
* What have we already tested?
* What worked?
* What failed?
* What is better than our current approach?
* What is not worth switching to?
* What should we watch but not adopt yet?
* What would adoption cost?
* What would break if we switched?
* What North Star goal would this support?

## 3. Role in the Ledger System

The Scout Ledger sits between Open Problems, Decisions, and Connections.

```text
Open Problems Ledger
  └── What problem needs a better solution?

Scout Ledger
  └── What options are we researching?

Decisions Ledger
  └── What did we decide to adopt, reject, or defer?

Connections Ledger
  └── What external tools/services are now dependencies?

Reconstruction Ledger
  └── How do we rebuild or integrate the adopted tool?
```

Important flow:

```text
Problem discovered
→ scout options
→ evaluate
→ decide
→ integrate
→ record dependency
→ record rebuild recipe
```

The Scout Ledger is not the final decision.
It feeds decisions.

---

## 4. What This Ledger Owns

## Owns

* tool/model/API/framework research
* comparison notes
* candidate evaluations
* adoption criteria
* rejection reasons
* watchlist items
* experiments and test results
* tool-switching cost analysis
* research links or references
* recommendation status
* revisit dates
* adoption readiness

## Does not own

* final architectural decisions
* all external dependency status
* implementation recipes
* billing/credential maps after adoption
* every article/link read
* every passing curiosity
* local one-off notes with no project impact

Those belong in Decisions, Connections, Reconstruction, Communications, and Open Problems ledgers.

The Scout Ledger should track **evaluated options**, not every interesting thought.

## 7. Scout Status Labels

Standardize statuses:

```markdown
## 3. Scout Status Labels

Use one:

- `queued`: worth evaluating later
- `researching`: actively being researched
- `testing`: hands-on test in progress
- `watch`: promising but not ready / not urgent
- `recommended`: should be adopted or tried
- `adopted`: moved into project
- `rejected`: not worth using
- `deferred`: useful, but not now
- `superseded`: replaced by better option
```

Optional fields:

```text
urgency: now / soon / later / watch
confidence: high / medium / low
switching_cost: low / medium / high
risk: low / medium / high
```

## 8. Evaluation Criteria

Every scout should be judged against project needs.

Suggested criteria:

```markdown
## 4. Evaluation Criteria

Evaluate candidates against:

### Fit

Does it solve a real project problem?

### Continuity

Does it support GitHub/file-tree continuity, or does it create hidden state?

### Inspectability

Can agents and humans understand what it is doing?

### Source-of-truth compatibility

Does it preserve source-of-truth discipline?

### Locality

Does it support Box/local-directory workflows?

### Safety

Could it trigger unsafe writes, sends, data leaks, or customer-facing actions?

### Integration cost

How hard is it to adopt?

### Switching cost

What breaks if we move from current system?

### Rebuildability

Can we recreate this setup later?

### Billing / dependency risk

Does it add paid, quota, credential, or uptime risk?

### North Star alignment

Which North Star goal does it support?
```

A candidate should not be adopted just because it is impressive.

It must fit the architecture.

### SCOUT-2026-04-28-001 — Approval Card / Risk Review UI Pattern

```markdown
## SCOUT-2026-04-28-001 — Approval Card / Risk Review UI Pattern

Status: queued  
Category: workflow / UI / safety  
Problem it addresses: risky moves can hide inside batch approval  
Related open problem: PROB-2026-04-28-003  
Related North Star goals: NS-04, NS-05, NS-06  
Urgency: soon  

### Question

What is the best UI/workflow pattern for isolating high-risk approvals?

### Current System

Risky moves may appear inside batch review. User may approve quickly while rushed.

### Candidate

Approval cards that require explicit confirmation based on risk class:

- `APPROVE FEE WAIVER`
- `APPROVE ENRICHMENT USE`
- `APPROVE PRICE COMMITMENT`
- `APPROVE SCOPE PROMISE`

### Why It Might Help

It preserves automation speed while making dangerous moves visible.

### Risks / Costs

Too much friction may slow routine work. Too little friction fails the purpose.

### Test Plan

Prototype card schema and review against Brenda & Steve case.

### Recommendation

Likely adopt after design.
```

````markdown
## SCOUT-2026-04-28-002 — Transcript Import / Fact Provenance Pattern

Related open problems: PROB-2026-04-28-001, PROB-2026-04-28-005

Urgency: soon  

### Question

What is the best structure for storing full transcripts and extracting allowed-to-use facts with provenance?

### Candidate

```text

Extract facts into:

```text

### Why It Might Help

It makes customer-facing fact use safe and auditable.

### Risks / Costs

May create too much per-client overhead if schema is overbuilt.

### Test Plan

Try with one high-value box, likely Brenda & Steve or Hugo.

### Recommendation

Likely adopt after Source-of-Truth / Allowed-To-Know design.

````

## 15. Tool Switching Rules

This section is extremely important.

```markdown
## 11. Tool Switching Rules

Do not switch tools just because a new tool looks better.

Before adopting a new tool, answer:

- What exact problem does it solve?
- What current system does it replace or augment?
- What North Star goal does it support?
- What new dependency does it create?
- What credentials, billing, or uptime risks come with it?
- What migration cost exists?
- What breaks if it fails?
- Can we revert?
- Does it preserve GitHub/file-tree source-of-truth?
- Does it create hidden state?
```

Adoption should require a decision.

```markdown
A scout recommendation is not an adoption decision.

Adoption requires:
- Decision Ledger entry
- Connections Ledger update if external
- Reconstruction recipe if complex
- Settings Ledger update if configurable
- Done Gate satisfied
```

## 18. Relationship to Open Problems Ledger

The Scout Ledger often starts with an open problem.

Example:

```text
Open Problem:
Allowed-to-know constraint not implemented.

Scout:
Evaluate transcript fact provenance patterns.

Decision:
Adopt Client Box-level `allowed_to_use.json`.
```

A scout without a problem or opportunity can still exist, but it should justify why it matters.

---

## 19. Relationship to Decisions Ledger

Scouting does not decide.

It recommends.

When a scout results in adoption or rejection with future impact, create a Decision record.

Example:

```text
Scout recommendation:
Use Mermaid first.

Decision:
Mermaid is first-wave visual format for ledgers.
```

Rejected choices can also become decisions if they keep coming back.

---

## 20. Relationship to Connections Ledger

If a scout adds an external dependency, update Connections.

Example:

```text
Scout:
Evaluate Supabase.

If adopted:
Connections Ledger records Supabase service, credentials, failure modes, billing risk.
```

Do not add tools to Connections just because they were scouted.

Only add active, planned, or watch dependencies with clear status.

---

## 21. Relationship to Reconstruction Ledger

If a scout becomes an implementation pattern, add reconstruction memory.

Example:

```text
Scout:
Approval-card pattern.

If adopted:
Reconstruction Ledger records how to rebuild approval-card workflow.
```

This turns research into reusable build knowledge.

---

## 22. Relationship to North Star

Every scout should name which North Star goal it supports.

If a scout does not support a North Star goal, it may still be useful, but the reason should be explicit.

This protects against tool-chasing.

Example:

```text
Mermaid visual maps
Supports:
NS-08 Visual Comprehension
NS-09 Agent Handoff Continuity
NS-10 Defense as Build Work
```

## 23. Scout Change Done Gate

Update this ledger when:

```markdown
- a new tool/model/API/framework is worth evaluating
- a research question is opened
- a test is run
- a candidate is rejected
- a candidate is recommended
- a candidate is adopted
- a candidate is put on watch
- a tool choice affects dependencies, settings, or source-of-truth
- a repeated research question comes back
```

Do not update for every random interesting link.

A scout entry should have project relevance.

## 25. Quality Bar

The Scout Ledger succeeds if future agents can answer:

* What are we considering?
* Why are we considering it?
* What problem does it solve?
* What did we already test?
* What did we reject?
* What should we watch?
* What should we adopt?
* What would adoption cost?
* What North Star goal does it support?
* What new dependency would it create?

It fails if:

* it becomes a pile of shiny links
* it does not tie options to real problems
* it does not record rejection reasons
* it recommends adoption without decisions
* it ignores dependency/billing/security risk
* it repeats research already done
* it causes tool churn

The Scout Ledger should make exploration disciplined and cumulative.

---

## 26. Final Operating Rule

The Scout Ledger should end with this:

```markdown
Curiosity is good.

Unremembered curiosity is churn.

Scout the option.
Record the lesson.
Adopt only when it serves the system.
```
