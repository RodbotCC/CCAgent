# Temporal Continuity Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Temporal Continuity Ledger** is the project’s living “what is true right now?” memory.

The Global Ledger explains the whole world of the project.

The Temporal Continuity Ledger explains the **current moment** inside that world.

It tracks what recently changed, what is currently active, what assumptions are carrying forward, what risks are open, what should not be forgotten, and what context future agents need so they do not restart from zero.

This is one of the project’s core anti-amnesia ledgers.

The uploaded ledger plan describes it as the ledger that preserves current state, active assumptions, recent changes, important context, and continuity between sessions. It should answer what changed recently, what is true now, what needs to carry forward, and what would be expensive or dangerous to rediscover. 

The Temporal Continuity Ledger is not a changelog.

It is a **handoff spine across time**.

---

# 1. Core Operating Principle

The Temporal Continuity Ledger should enforce this rule:

> Every meaningful work session should leave the next session less blind.

The project should never rely on a future model remembering the chat.

It should not rely on local-only notes.

It should not rely on a human re-explaining the whole context every time.

If a fact, warning, assumption, decision-in-motion, active task, or fragile context matters to the next session, it belongs in the Temporal Continuity Ledger or in a linked local continuity file.

The goal is not to record everything.

The goal is to preserve what would be costly, risky, or annoying to rediscover.

The Temporal Continuity Ledger sits directly under the Global Ledger.

```text
Global Ledger
  └── What is this project?
  └── What are the major systems?
  └── What rules govern the whole build?

Temporal Continuity Ledger
  └── What is true right now?
  └── What just changed?
  └── What is active?
  └── What must carry forward?
  └── What should the next agent know before continuing?
```

If the Global Ledger is the constitution, the Temporal Continuity Ledger is the current field report.

# 4. What This Ledger Owns

The Temporal Continuity Ledger owns the current working state across sessions.

## Owns

* current active project state
* recent meaningful changes
* active workstreams
* current operating assumptions
* active risks that need near-term attention
* current human instructions/preferences
* session handoff notes
* current blockers
* pending next actions
* “do not forget” warnings
* recent audit findings that affect the next steps
* state transitions between phases or workflows
* cross-system context that is not local to one folder

## Does not own

* every historical change
* full decision rationale
* detailed local file maps
* permanent architecture rules
* full open-problem inventory
* implementation recipes
* page/widget ownership details
* exhaustive task tracking

Those belong elsewhere:

```text
Decisions Ledger          → why choices were made
Open Problems Ledger      → known unresolved problems
Audit Ledger              → formal audit findings
Directory Ledger          → file tree shape
Page / Widget Ledgers     → local UI ownership
Reconstruction Ledger     → how something was built
Global Ledger             → durable project-wide identity/rules
```

The Temporal Continuity Ledger should stay light enough to read before every session.

## 7.5 Active Assumptions

Purpose: capture assumptions currently guiding the work.

This is critical because assumptions silently shape agent behavior.

Example:

```markdown
## 4. Active Assumptions

- GitHub should be treated as durable source of truth.
- Local machine may contain unpushed work, so direct GitHub writes require caution.
- The project prefers FileTree orchestration over RAG-style memory.
- Boxes are the preferred local memory structure.
- Seven-day plans are strategy drafts, not unconditional send licenses.
- Client replies invalidate or at least pause existing plans until reviewed.
- Enrichment is allowed for internal strategy but must not leak into customer-facing copy unless grounded in comms or explicitly approved.
- Comeketo currently wants visible automation movement, but the user wants to keep the system safer and more controlled.
- Early automation should use boring-safe movement before aggressive rescue plays.
```

Assumptions should be reviewed often.

If an assumption becomes a decision, move it to the Decisions Ledger.

If an assumption becomes wrong, update it immediately.

## 7.6 Carry-Forward Context

Purpose: preserve the stuff the next session really needs.

This section is different from active assumptions. It is more conversational, memory-like, and context-heavy.

Example:

```markdown
## 5. Carry-Forward Context

The user is trying to build a durable GitHub-backed memory system where agents can continue work without relying on chat history. They care deeply about ledgers, boxes, visual maps, and source-of-truth clarity.

The user is under pressure to show Comeketo that automation is moving, but they are worried the system may act too aggressively. They want a posture that preserves momentum while reducing unsafe autonomy.

The Brenda & Steve case is the key lesson:
- The fee-waiver move may have been strategically correct.
- The problem was that the risk was not isolated enough during batch approval.
- Future risky moves should become explicit approval cards, not hidden inside normal batch approvals.

The user wants to proceed through ledger design one at a time. When the user types `P`, produce the next ledger outline.
```

This is where you preserve the “vibe that matters,” but in a disciplined way.

## 7.7 Current Risks / Fragile Areas

Purpose: make near-term risks impossible to miss.

Example:

```markdown
## 6. Current Risks / Fragile Areas

### Risk: Plan / guardrail contradiction

Some plans may contain copy written before guardrails were enforced. Audit each plan before trusting it.

### Risk: Batch approval hides dangerous moves

Fee waivers, discounts, pricing claims, guest-count promises, and enrichment-based personalization should require isolated approval.

### Risk: Reply invalidates plan

If a lead replies, the plan should be paused and possibly rewritten.

### Risk: Enrichment boundary

Client profiles may contain useful but non-customer-facing facts. The system needs an allowed-to-know / allowed-to-say layer.

### Risk: GitHub writes vs local work

The human may have unpushed local work. Direct GitHub commits can create rebase/merge complexity.

### Risk: Generated UI confidence

The Boxes page may display old or partial source data. It should be treated as a view, not as the canonical truth.
```

This section can later feed the Open Problems Ledger.

The Temporal version should list what matters **right now**.

## 7.13 Update Rules

Purpose: define when to update this ledger.

Example:

```markdown
## 12. Update Rules

Update this ledger when:

- a meaningful work session ends
- current project state changes
- active workstreams change
- a major risk becomes active or is resolved
- user gives a durable instruction/preference
- a new carry-forward assumption appears
- a box/page/system is recently touched in a way future agents should know
```

Also define when **not** to update:


# 9. Temporal Continuity Update Workflow

At the end of a meaningful session, an agent should run this mental checklist:

```text
Did the current project state change?
Did the active workstream change?
Did the user give a preference that should carry forward?
Did we touch a system future agents should know about?
Did we discover a risk?
Did we leave something unfinished?
Did we create a handoff instruction?
```

If yes, update the Temporal Continuity Ledger.

Suggested end-of-session protocol:

```text
1. Update Current Snapshot if needed.
2. Update Active Workstreams.
3. Add Recent Meaningful Changes.
4. Add or remove Active Assumptions.
5. Add Carry-Forward Context.
6. Add Current Risks if needed.
7. Update Next Agent Handoff.
8. Append short Session Log entry.
9. Commit changes.
```

This should take minutes, not hours.

The Temporal Continuity Ledger should be fast enough that agents actually use it.

# 10. Relationship to the Global Ledger

The Global Ledger should link to Temporal Continuity as:

```markdown
Temporal Continuity Ledger:
Current project moment, active assumptions, recent changes, active risks, and next handoff.
```

The Temporal Continuity Ledger should link back to Global Ledger as:

```markdown
Read this after the Global Ledger. If the Global Ledger tells you what this project is, this file tells you where the project currently is in time.
```

If they conflict:

* Global Ledger wins for permanent project identity and rules.
* Temporal Continuity wins for current working state.

Example:

```text
Global Ledger says:
Client Boxes are canonical per-client memory.

Temporal Continuity says:
Currently auditing Client Boxes one at a time, Brenda & Steve completed, Hugo likely next.
```

Both are true. They operate at different altitude.

# 11. Relationship to Boxes

Each important Box can have its own local temporal continuity file.

Example:

```text
Auto/Client Boxes/Brenda & Steve/
  client_ledger.md
  2026-04-28_audit_marker.md
  BOX_CONTINUITY.md       optional later

Auto/orchestrator/
  BOX_CONTINUITY.md       optional later
```

Global temporal continuity should not duplicate every local detail.

Instead:

```markdown
Recently touched: Brenda & Steve box. See local audit marker.
```

The pattern should be:

```text
Project Temporal Continuity
  └── remembers what areas changed and why

Local Box Continuity
  └── remembers local state, local risks, local handoff
```

This protects the global ledger from becoming too heavy.

# 12. Relationship to Client Boxes

For Client Boxes specifically, temporal continuity matters a lot because client state changes over time.

Every client box eventually needs its own state rhythm:

```text
Comms changed
  → state updates
  → plan validity checked
  → allowed-to-know facts updated
  → next move recalculated
  → approval requirements checked
  → audit marker/handoff left if touched
```

The global Temporal Continuity Ledger should only track cross-client context, such as:

```markdown
The Client Box audit workflow is active.
Brenda & Steve was cleaned first.
Future boxes should be flattened for safe movement.
Allowed-to-know constraints need implementation.
```

Individual client details belong inside their boxes.

# 13. Relationship to GitHub / Local Drift

The Temporal Continuity Ledger should explicitly record Git workflow state when relevant.

Because agents may write to GitHub while the user has local unpushed work, the ledger should have a small current Git posture section.

Example:

```markdown
## Current Git Posture

- Direct GitHub writes happened during Brenda & Steve cleanup.
- User may have local unpushed work.
- Before further direct GitHub writes, consider branch strategy or ask whether to continue direct-to-main.
- Avoid force-push assumptions.
```

This does not need to be permanent.

It should exist while the risk is active.

# 14. Relationship to “Sitting There”

This ledger should recognize that “nothing happened” can still be a state.

That matters for automation.

Example:

```markdown
## Inactivity as State

If a client, workstream, or system is intentionally waiting, record it as waiting.

Waiting is not absence.

Examples:
- waiting for client reply
- waiting for state sweep
- waiting for manual approval
- waiting for Git sync
- waiting for next ledger command
- waiting for full transcripts
```

This is a beautiful fit for your Box architecture.

Agents often mistake silence for missing context.

The Temporal Continuity Ledger should teach them that silence can be meaningful.

---

# 15. Relationship to Plans

The Temporal Continuity Ledger should encode this rule:

```markdown
Plans age.

Any plan that depends on time, client state, or external response has a freshness window.
```

For client plans:

```text
No reply:
  plan may remain valid if dates and guardrails still pass.

Reply received:
  plan is stale until reviewed.

New transcript imported:
  plan may need rewrite.

Guardrails changed:
  plan must be re-audited.

Approval status changed:
  future risky moves must be rechecked.
```

This prevents stale plans from quietly staying alive.

# 18. Quality Bar

The Temporal Continuity Ledger succeeds if a future agent can read it and quickly understand:

* what is happening now
* what just changed
* what workstreams are active
* what assumptions are guiding the work
* what risks are currently alive
* what the user wants next
* what should not be forgotten
* where to continue

It fails if:

* it becomes a full changelog
* it becomes stale
* it duplicates every other ledger
* it does not preserve active assumptions
* it hides current risks
* it omits user preferences
* it is too long to read at session start
* it does not tell the next agent what to do

The file should feel like opening the cockpit log before taking over the aircraft.

---

# 19. Final Operating Rule

The Temporal Continuity Ledger should end with this:

```markdown
The project should not wake up confused.

Every meaningful session should leave the next session with a clearer current state, fewer rediscovered problems, and a safer path forward.

If the project changed in time, record the change in time.
```
