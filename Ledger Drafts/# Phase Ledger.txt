# Phase Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Phase Ledger** tracks major phases of the project.

A phase is bigger than a task and smaller than the entire North Star.

It may be:

* a product milestone
* a build sprint
* a refactor pass
* a safety-hardening pass
* a release cycle
* a ledger implementation wave
* a Client Box cleanup wave
* an automation rollout stage
* a UI rebuild phase
* a source-of-truth migration

The uploaded ledger plan describes the Phase Ledger as the place to track what phase the project is in, the goal of that phase, what deliverables belong to it, what is in scope, what is out of scope, what is complete, what remains, what risks/blockers exist, and what ledgers must be updated before the phase closes. 

This matters because not every task should be judged in isolation.

Some work only makes sense as part of a larger phase.

The Phase Ledger gives the project milestone memory.

---

# 1. Core Operating Principle

The Phase Ledger should enforce this rule:

> A phase is not just a collection of tasks. It is a bounded chapter of the build.

A phase should tell future agents:

* What are we trying to accomplish right now?
* Why does this phase exist?
* What is in scope?
* What is out of scope?
* What deliverables define success?
* What has already been completed?
* What remains?
* What risks or blockers matter?
* Which ledgers/maps must be updated before the phase closes?
* What would make this phase complete enough to move on?

The Phase Ledger prevents “random progress.”

It gives the project chapters.

# 3. Role in the Ledger System

The Phase Ledger sits between North Star and daily work.

```text id="hul3y6"
North Star Ledger
  └── What are we trying to become?

Phase Ledger
  └── What chapter are we in right now?

Definition of Done Ledger
  └── What does complete mean?

Temporal Continuity Ledger
  └── What is true right now?

Open Problems Ledger
  └── What is blocking or risky?

Audit Ledger
  └── Did the phase meet its checks?
```

A task may support a phase.

A phase should support North Star goals.

The Phase Ledger helps prove that relationship.

---

# 4. What This Ledger Owns

## Owns

* phase list
* active phase status
* phase goals
* phase scope
* out-of-scope boundaries
* phase deliverables
* phase-level Definition of Done
* phase risks/blockers
* phase progress
* phase completion notes
* phase closeout requirements
* phase-to-ledger update requirements
* phase timeline
* phase dependencies

## Does not own

* every task detail
* every file description
* all open problems
* every current session note
* all audit details
* every decision rationale
* full implementation recipes

Those belong in Open Problems, Temporal Continuity, Audit, Decisions, Reconstruction, and local Box ledgers.

The Phase Ledger should answer “what chapter are we in?” not “what line changed?”

# 7. Phase Status Labels

```markdown id="6u2ayj"
## 3. Phase Status Labels

Use one:

- `planned`: phase is defined but not active
- `active`: currently being worked
- `paused`: intentionally waiting
- `blocked`: cannot proceed until blocker resolves
- `partial`: partially completed, not fully closed
- `closing`: closeout / verification in progress
- `complete`: phase met close criteria
- `superseded`: replaced by another phase
- `abandoned`: intentionally stopped
```

Optional fields:

```text id="3jdy4j"
priority: critical / high / medium / low
confidence: high / medium / low
```

# 15. Relationship to Temporal Continuity

Temporal Continuity says what is true right now.

Phase Ledger says what chapter the project is in.

Example:

```text id="qwsslh"
Temporal:
User is currently asking for one ledger at a time.

Phase:
Ledger System Buildout is active and outlines are being produced.
```

Temporal should reference active phase.

Phase should link to temporal status.

---

# 16. Relationship to North Star

Every phase should support at least one North Star goal.

If a phase does not map to a North Star, ask why.

Example:

```text id="gnisum"
Ledger System Buildout supports:
NS-01 Durable Project Memory
NS-02 FileTree Orchestration
NS-09 Agent Handoff Continuity
NS-10 Defense as Build Activity
```

This protects against busywork phases.

---

# 17. Relationship to Definition of Done

Each phase has its own Definition of Done.

The Definition of Done Ledger defines the general standard.
The Phase Ledger applies it to the milestone.

Example:

```text id="w6k9hr"
General DoD:
Update relevant ledgers.

Phase DoD:
Ledger System Buildout is not done until Global, Temporal, Index, visuals, and first audit exist.
```

---

# 18. Relationship to Open Problems

Phases should not hide unfinished work.

If a phase closes with remaining issues, those issues must move to Open Problems or a next phase.

Example:

```text id="b8rjki"
Client Box Safety Hardening closes, but approval UI not implemented.
→ Open Problem remains or new Approval Workflow Phase begins.
```

No unresolved problem should vanish at phase close.

---

# 19. Relationship to Audit Ledger

Every meaningful phase should end with an audit.

Example:

```text id="f7sqoz"
Phase:
Ledger System Buildout

Closing audit:
Do ledgers exist?
Are links valid?
Is INDEX current?
Do update rules exist?
Are visuals present?
```

The Audit Ledger records the evidence.

---

# 20. Relationship to Reconstruction Ledger

A phase that creates a repeatable pattern should update Reconstruction.

Example:

```text id="s32c0t"
Client Box Safety Hardening creates repeatable audit workflow.
→ Reconstruction Ledger records Client Box cleanup recipe.
```

This preserves how the phase was done.

---

# 21. Phase Change Done Gate

Update this ledger when:

```markdown id="8jebas"
- a new phase starts
- a phase scope changes
- deliverables change
- a phase is paused/blocked
- a phase closes
- a phase is superseded
- major risks/blockers change
- phase Done Gate changes
- phase work creates new open problems
```

Do not update for tiny task movement unless it affects phase progress.

# 23. Quality Bar

The Phase Ledger succeeds if future agents can answer:

* What phase are we in?
* Why does this phase exist?
* What is in scope?
* What is out of scope?
* What deliverables define success?
* What has been completed?
* What remains?
* What risks/blockers exist?
* What ledgers must be updated before close?
* What phase comes next?

It fails if:

* it becomes a giant task list
* phases do not have close criteria
* active phases are stale
* out-of-scope boundaries are missing
* phase risks are hidden
* phase closeout does not update other ledgers
* work cannot be tied to North Star goals

The Phase Ledger should make project momentum legible.

The Phase Ledger should end with this:

```markdown id="i01sr8"
A phase is a chapter of the build.

Finish the chapter.
Carry the unfinished truth forward.
```
