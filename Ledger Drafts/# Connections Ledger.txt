# Connections Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Connections Ledger** tracks every external system the project depends on.

This includes applications, APIs, services, credentials, MCP servers, model providers, databases, billing dependencies, automation platforms, GitHub connectors, Close, Slack, Google Drive, ClickUp, Supabase, paid tools, local servers, and anything else that can make the project work — or break.

The uploaded ledger plan describes this ledger as part technical map and part operations ledger. It should answer what external apps the project uses, which services are required or optional, what APIs/credentials are involved, what tools are paid, when billing or tokens renew, what breaks if a service goes down, and who depends on each connection. 

This ledger prevents surprise breakage.

A project like this does not only fail because code is wrong.

It can fail because:

* an API key expires
* a token is missing
* a paid service runs out
* a connector is not enabled
* a local server is down
* a model setting changed
* a webhook points to the wrong place
* Close, Slack, GitHub, or Google Drive state is misunderstood
* one automation assumes a connection exists that is only configured locally

The Connections Ledger is the project’s dependency reality check.

---

# 1. Core Operating Principle

The Connections Ledger should enforce this rule:

> If the project depends on an outside system, that dependency must be visible.

Not necessarily the secret itself.

Never store raw secrets in the ledger.

But the dependency should be recorded:

* service name
* purpose
* where credentials are expected
* whether it is required or optional
* who/what uses it
* what breaks if unavailable
* how to verify it
* renewal/billing/trial risk if relevant
* local vs GitHub vs cloud state
* owner or responsible human/system

The project should never be surprised by a forgotten dependency.

# 3. Role in the Ledger System

The Connections Ledger sits across the whole project because external dependencies cut through everything.

```text id="d31t4c"
Global Ledger
  └── project-wide orientation

Connections Ledger
  └── external systems, APIs, tools, credentials, services, billing, failure modes

Settings Ledger
  └── user-configurable settings and credential surfaces

Asset / Widget Map
  └── which pages/widgets call which APIs/services

Source-of-Truth Ledger
  └── which external system owns which data

Automation Ledgers / Boxes
  └── which automations depend on which services
```

The Connections Ledger is not just for developers.

It is for operators too.

It should answer:

> “What do we rely on, and what happens if it breaks?”

---

# 4. What This Ledger Owns

## Owns

* external service inventory
* connector inventory
* API dependency map
* credential expectation map
* required vs optional service classification
* billing/trial/renewal notes
* failure modes
* verification steps
* local/cloud dependency distinctions
* service owners
* downstream consumers
* replacement or fallback notes
* security/sensitivity notes

## Does not own

* raw API keys
* passwords
* OAuth tokens
* full endpoint documentation
* every local environment variable value
* all source-of-truth rules
* every page API call
* all settings UI details

Those belong in secure stores, `.env`, Settings Ledger, Source-of-Truth Ledger, API docs, and app configuration.

The Connections Ledger should say **what exists and where to verify**, not expose secrets.

# 6. Security Rules

This must be near the top.

```markdown id="nytbqd"
## 2. Security Rules

Never store raw secrets in this ledger.

Do not record:
- API keys
- access tokens
- refresh tokens
- passwords
- private keys
- OAuth secrets
- service account JSON contents
- database passwords

Allowed:
- service name
- credential variable name
- expected storage location
- owner/responsible party
- whether credential exists
- how to verify connection safely
- expiration/renewal notes
- what breaks if missing

If a credential is missing, record the missing dependency, not the secret.
```

This keeps the ledger safe while still useful.

# 8. Required Connections

Purpose: what must exist for core operation.

Example:

```markdown id="g19e26"
## 4. Required Connections

| Connection | Required For | What Breaks If Missing | Verification |
|---|---|---|---|
| GitHub | durable source of truth, agent handoff, repo memory | agents lose shared canonical continuity | can fetch/push repo |
| Close API | lead comms, tasks, sends, activity sweeps | no live CRM state, no send/sweep behavior | safe status/activity test |
| Slack | Andre review surfaces, team audit reporting | review/approval loop breaks | test channel/DM post or status check |
| Local `.env` | credentials/config | APIs and external actions fail | server status endpoint or env check |
| App server | UI/API runtime | pages cannot load dynamic data | `/api/status` or local health check |
```

Do not pretend every service is required.

Separate required from optional so failure response is clear.

This helps triage.

If optional service fails, the system can continue with reduced capability.

# 14. Verification Playbook

Purpose: safe checks for connections.

```markdown id="d0hxfl"
## 10. Verification Playbook

### Safe read checks

Use safe read checks before write checks.

Examples:
- fetch known GitHub file
- fetch Close lead metadata
- fetch Close recent activities without sending
- check `/api/status`
- check `/api/pieces/status`
- list workflows/triggers without saving
- read Slack channel/user identity without posting if possible

### Write checks

Only perform write checks when explicitly authorized.

Examples requiring caution:
- Slack test post
- Close note creation
- workflow save
- trigger save
- credential save
- customer-facing send

### Never use as a connection test

- sending a customer SMS
- sending a customer email
- changing lead status
- moving tasks in production
- deleting workflows/triggers
- overwriting credentials
```

This protects against “testing” by causing damage.

# 16. Connection Change Done Gate

Purpose: define when this ledger must be updated.

```markdown id="7n9hbp"
## 12. Connection Change Done Gate

Update this ledger when:

- a new external service is added
- a service is removed
- an API dependency changes
- a credential variable is renamed
- a service changes from optional to required
- a service changes from required to optional
- a new side effect is introduced
- billing/quota/trial risk is discovered
- verification steps change
- a failure mode is discovered
- a local-only dependency becomes important
- an automation starts depending on a connection
```

For UI changes:

```markdown id="pw3bpp"
If a page or widget begins using a new external connection, update both the Connections Ledger and the Asset / Widget Map.
```

For settings changes:

```markdown id="n1j5qv"
If a credential becomes user-configurable, update the Settings Ledger too.
```

```json id="cvhxxm"
      "data_written": ["sms/email activities if sending", "task movements", "notes"],
      "side_effect_risk": "very_high",
      "failure_mode": "No live comms import, no sends, stale client state.",
      "verification": ["safe read-only lead/activity fetch"],
      "fallback": ["manual Close review", "use last synced comms marked stale"]
    }
  ],
  "failure_modes": [
    {
      "symptom": "Client ledgers stale",
      "likely_connections": ["close", "cowork_scheduled_tasks"],

    }
  ]
}
```

This JSON can later power a connection health dashboard.

# 19. Relationship to Settings Ledger

Connections and settings overlap, but they are not the same.

```text id="kut93x"
Connections Ledger:
What external systems exist and what depends on them.

Settings Ledger:
What configurable options expose or control those systems.
```

Example:

```text id="zz2wlg"
Connections Ledger:
Close API exists and is critical.

Settings Ledger:
Where Close API key is configured, whether user can edit it, default behavior, masking rules.
```

When credentials become UI-editable, update both.

# 20. Relationship to Source-of-Truth Ledger

Connections can be sources of truth, but not always.

Example:

```text id="3htmg1"
Close:
source of truth for live lead activity and CRM status.

GitHub:
source of truth for project memory/code.

Slack:
communication/reporting surface, not source of client truth unless a specific approval event is recorded.

Google Drive:
source of internal docs if linked; may not be canonical if docs are mirrored into GitHub.
```

The Source-of-Truth Ledger will define authority.

The Connections Ledger defines dependency.

# 21. Relationship to Asset / Widget Map

Every widget with external dependencies should point to the Connections Ledger.

```text id="5li3om"
  → Connections Ledger for services
  → Settings Ledger for UI/config behavior

Activity page Pieces panel
  → Connections Ledger for Pieces dependency

  → external channel dependencies depending on target
```

# 22. Relationship to Client Boxes

Client Boxes depend on external connections indirectly.

```text id="474fvs"
Close → comms/state truth
Slack → approval/alerts
Cowork → scheduled fires/sweeps
GitHub → durable box memory
OpenAI/Claude → audit/plan generation
```

A Client Box can look healthy in GitHub but still be stale if Close/Cowork sweeps are not running.

So the Connections Ledger should help agents distinguish:

```text id="adgh1h"
Box files exist
vs
Box state is fresh
```

Freshness depends on connections.

# 24. Quality Bar

The Connections Ledger succeeds if future agents and humans can answer:

* What external services does the project depend on?
* Which are required?
* Which are optional?
* What credentials are expected?
* Where should credentials live?
* What breaks if a service fails?
* How can the connection be safely verified?
* What services are paid/quota-sensitive?
* Which pages/widgets/automations depend on each service?
* Which connections are local-only vs cloud/shared?
* What fallback exists?

It fails if:

* it stores secrets
* it invents dependencies
* it ignores billing/quota risk
* it does not distinguish required from optional
* it does not include failure modes
* it lacks verification steps
* it becomes stale when services change
* it does not connect to settings, widgets, or source-of-truth ledgers

This ledger should make the project operationally safer.

# 25. Final Operating Rule

The Connections Ledger should end with this:

```markdown id="ihe5pq"
External systems are part of the architecture.

If the project needs it, map it.
If it can break the project, name the failure mode.
If it requires a secret, record the location — never the secret.
```
