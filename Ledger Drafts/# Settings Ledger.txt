# Settings Ledger — Project-Wide Implementation Outline

## 0. Purpose

The **Settings Ledger** tracks the knobs, switches, credentials, runtime options, user preferences, environment variables, feature flags, model/provider choices, UI preferences, and configurable behaviors that affect the project.

Settings are dangerous because they often look small.

A single setting can change:

* what model is used
* whether a workflow runs
* whether credentials are available
* whether a page writes to disk
* whether a widget can submit a delegation
* whether Close/Slack/Pieces/OpenAI works
* whether customer-facing automation is active
* whether generated state is stale or refreshed
* whether a local run behaves differently from GitHub/cloud

The Settings Ledger exists so future agents can answer:

> What can be configured, where is it configured, what does it affect, and what breaks if it changes?

---

# 1. Core Operating Principle

The Settings Ledger should enforce this rule:

> A setting is behavior disguised as configuration.

Settings are not passive.

They can change the project’s runtime behavior without changing code. So they need memory.

A future agent should be able to answer:

* What settings exist?
* Where are they stored?
* Are they user-facing or internal?
* Are they local-only or GitHub-backed?
* Are they safe to commit?
* Do they affect credentials?
* Do they affect model/provider behavior?
* Do they affect automation?
* Do they affect sends or external actions?
* What pages/widgets depend on them?
* What should be verified after changing them?

The Settings Ledger is the project’s configuration map.

# 3. Role in the Ledger System

The Settings Ledger overlaps with Connections, Source-of-Truth, Pages, Widgets, and Automation.

```text
Connections Ledger
  └── What external systems exist?

Settings Ledger
  └── What configuration controls those systems?

Source-of-Truth Ledger
  └── Which settings source is authoritative?

Page Ledger
  └── Which pages expose settings?

Widget Ledger
  └── Which widgets read/write settings?

Definition of Done Ledger
  └── What must be updated when settings change?
```

Example:

```text
Connections Ledger:
Close is a required CRM dependency.

Settings Ledger:
Close API key is expected in `.env`, must never be committed, and changes affect comms sweeps/sends.
```

# 4. What This Ledger Owns

## Owns

* setting registry
* configuration source locations
* environment variable names, never values
* user-facing settings
* internal runtime settings
* model/provider settings
* credential settings
* feature flags
* UI preferences
* localStorage settings
* `.env` expectations
* settings pages/widgets
* setting side effects
* setting verification steps
* safe defaults
* settings change Done Gate

## Does not own

* raw secrets
* API keys
* passwords
* OAuth tokens
* full external service dependency details
* every application constant
* all source-of-truth rules
* every UI component affected by a setting

Those belong in secure stores, Connections Ledger, Source-of-Truth Ledger, File Contents Ledger, and code.

This ledger records configuration, not secrets.

# 6. Security Rules

This section must be near the top.

```markdown
## 2. Security Rules

Never store raw secrets in this ledger.

Do not record:

- API keys
- passwords
- OAuth tokens
- refresh tokens
- private keys
- service account JSON contents
- database passwords
- raw `.env` values

Allowed:

- setting name
- environment variable name
- expected storage location
- service affected
- whether a value is required
- whether it is user-facing
- how to verify safely
- what breaks if missing
- whether it is safe to commit
```

Include this rule:

```markdown
If a setting is secret-bearing, the ledger may describe the key but must not contain the value.
```

Use stable IDs.

```text
SET-<category>-###
```

Examples:

```text

SET-MODEL-001
SET-AUTO-001
SET-UI-001
```

Template:

```markdown
## SET-<CATEGORY>-### — <Setting Name>

Status:
Category:
Scope:
Storage:
User-facing:
Secret-bearing:
Safe to commit:
Owner:
Used by:
Related connections:
Related pages/widgets:
Related ledgers:

### Purpose

What this setting controls.

### Expected Location

Where it lives.

### Allowed Values / Shape

What values are valid. Do not include secrets.

### Default / Fallback

What happens if missing.

### Side Effects

What changes when this setting changes.

### Verification

How to safely verify it.

### Failure Mode

What breaks if wrong/missing.

### Change Rules

What must be updated or checked after changing it.
```

This makes settings auditable.

# 10. Credential / Secret Settings

This section should be very careful.

## 10.1 Close API Key

```markdown
## SET-CRED-001 — Close API Key

Status: needs verification / likely required
Category: credential
Scope: runtime / CRM
Storage: `.env` or secure runtime config
User-facing: maybe through settings UI, verify
Secret-bearing: yes
Safe to commit: no
Owner: Comeketo / operator
Used by:
- comms_state_sweep
- Close reads/writes
- inbox automation
- scheduled fires
- lead/task state

### Purpose

Authorizes access to Close CRM data and actions.

### Expected Location

`.env` or secure server config. Exact variable name must be verified from code.

### Allowed Values / Shape

Secret token/string. Do not record value.

### Default / Fallback

If missing, Close reads/writes fail. Client Boxes may become stale. Customer-facing sends must not run.

### Side Effects

Changing this setting can alter which Close account/workspace is used.

### Verification

Use safe read-only Close API check. Never test by sending customer-facing SMS/email.

### Failure Mode

- no live comms import
- no state sweep
- no sends
- stale Client Boxes
- reply gate unreliable

### Change Rules

Update Connections Ledger if credential storage or service status changes.
```

# 12. Automation Settings

This project needs explicit automation settings because of customer-facing risk.

## 12.1 Client Cadence Active Flag

```markdown
## SET-AUTO-001 — Client Cadence Active Flag

Status: active
Category: automation/client-state
Scope: per-client
Storage: `Auto/Client Boxes/<Name>/00_meta.json`
User-facing: no
Secret-bearing: no
Safe to commit: yes, but sensitive operational state
Used by:
- scheduled fires
- ready checks
- client audits
- state sweep

### Purpose

Indicates whether a client box has an active cadence.

### Expected Location

`00_meta.json`, likely field `active_cadence`.

### Side Effects

If true, automation may consider future scheduled moves. If false, client should not be actively touched by cadence unless manually directed.

### Verification

Check `00_meta.json`, plan, comms, and client ledger together. Do not trust flag alone.

### Failure Mode

- active false when client should move
- active true when cadence should be paused
- scheduled fire risk if stale

### Change Rules

Any manual change should leave audit/handoff note.
If client replied, cadence should pause or plan should be marked stale according to future rewrite rule.
```

# 16. Settings Verification Playbook

```markdown
## 12. Settings Verification Playbook

### Safe verification

Prefer:

- status endpoint checks
- read-only connection checks
- masked credential presence checks
- local config parse checks
- dry-run behavior checks
- non-customer-facing model test

### Risky verification

Requires explicit authorization:

- posting to Slack
- writing credentials
- clearing credentials
- saving workflow/trigger settings
- changing active automation flags
- sending customer-facing SMS/email

### Never use as a settings test

- customer-facing send
- deleting production workflow
- clearing credential without confirmation
- changing Close lead status
- force-pushing config changes
```

This keeps debugging safe.

# 17. Settings Change Done Gate

```markdown
## 13. Settings Change Done Gate

A settings change is not done until:

- [ ] setting owner is clear
- [ ] storage location is clear
- [ ] secret-bearing status is clear
- [ ] safe-to-commit status is clear
- [ ] affected pages/widgets are known
- [ ] affected connections are known
- [ ] side effects are understood
- [ ] fallback/default behavior is defined
- [ ] verification is performed or honestly marked unavailable
- [ ] Settings Ledger updated
- [ ] Connections Ledger updated if external service changed
- [ ] Source-of-Truth Ledger updated if authority changed
- [ ] Page/Widget Ledgers updated if UI changed
```

For secret settings:

```markdown
No settings change is done if raw secrets are exposed in committed files, ledgers, screenshots, logs, or chat artifacts.
```

For automation settings:

```markdown
No customer-facing automation setting change is done unless send/approval/dry-run behavior is verified.
```

# 20. Relationship to Connections Ledger

Connections tells the project what services exist.

Settings tells the project what config controls those services.

Example:

```text
Connections Ledger:
Slack is used for alerts.

Settings Ledger:
Slack token/channel IDs are expected in secure config and must not be committed.
```

If a service is added, update Connections.
If its credentials/config are exposed to the app, update Settings too.

---

# 21. Relationship to Source-of-Truth Ledger

Some settings are source-of-truth for behavior.

Example:

```text
active_cadence in `00_meta.json` may be source-of-truth for whether a client cadence is active.

But it is not sufficient alone to authorize sending.
```

The Source-of-Truth Ledger should define authority.

The Settings Ledger should define configuration mechanics.

# 26. Quality Bar

The Settings Ledger succeeds if future agents can answer:

* What settings exist?
* Where are they stored?
* Are they secret-bearing?
* Are they safe to commit?
* What systems depend on them?
* What pages/widgets edit them?
* What happens if they are missing?
* How do we verify them safely?
* What settings affect automation?
* What settings affect customer-facing actions?
* What settings need decisions or implementation?

It fails if:

* it stores secrets
* it invents variable names
* it ignores side effects
* it does not distinguish localStorage from `.env` from GitHub-backed config
* it does not connect to Connections
* it does not identify customer-facing automation settings
* it lets settings changes happen without verification
* it becomes stale after settings UI changes

This ledger should make configuration visible, safe, and auditable.

# 27. Final Operating Rule

The Settings Ledger should end with this:

```markdown
A setting is not just a value.

It is a behavior switch.

Know where it lives.
Know what it changes.
Know what breaks when it is wrong.
Never write down the secret.
```
